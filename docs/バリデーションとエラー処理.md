# バリデーションとエラー処理の設計

フォーム入力の検証と、サーバーエラーの伝え方・出し方の方針。
（最終更新: 2026-07-01）

---

## 0. 大原則

> **クライアント検証＝UX（即時フィードバック）。サーバー検証＝本当の門番。**
> 両方やる。役割が違うので重複ではなく補完。

- クライアント検証は**迂回可能**（DevTools・直接API）。安全をここに依存しない。
- 最終防衛線は必ずサーバー（tsoaの型検証＋DB制約）。

---

## 1. 誰が何を捕まえるか（分担）

| 種類 | 担当 | 例 |
|---|---|---|
| その場で分かる | **クライアント**（即時） | 必須漏れ・日付/数値の形式・文字数 |
| サーバーにしか分からない | **サーバー**（送信後） | ユーザー名の重複・権限・他データ整合・業務ルール |

クライアントはDBを見ないので「重複」は判定できない → サーバーが返す。
「必須が空」をサーバー往復で知るのは遅い → クライアントが即出す。

---

## 2. 出るタイミング（mode）

世の中の標準は「最初は静か → submitかblurで出す → 一度出たらリアルタイム」。

| mode | 挙動 |
|---|---|
| onSubmit | submitまで出さない → 以後リアルタイム |
| onBlur | 項目を離れた時 |
| **onTouched（採用）** | 触った項目だけ、以後リアルタイム。体感が良い |
| onChange | 打つたび（うるさい） |

→ 本PJのフォームは **`mode: 'onTouched'`**。開いた直後は赤字なし → 触る/submitで出る → 一度出たら直すたびに消える。

---

## 3. 1回のsubmitの流れ

```
submit
  │
  ▼
① クライアント検証(Zod) ── NG → 赤字、サーバーへ送らない（停止）
  │ OK
  ▼
② サーバー検証(tsoa)＋DB制約 ── NG → エラー返却 → 画面表示
  │ OK
  ▼
 成功（ダイアログを閉じてトースト）
```

クライアントを通る間はサーバーに届かない。だから段階的に出る（同時に二重で怒らない）。

---

## 4. サーバーエラーの伝え方（実装）

サーバーは「ステータス＋JSON」で返す → API関数が例外(`ApiError`)に翻訳 → 画面が表示。

```
サーバー                              クライアント
tsoa検証NG → 422 {message, details}
UNIQUE違反 → 409 {error}        →  api層: !res.ok なら ApiError(status, body) を throw
対象なし   → 404 {error}              ↓
                                  ダイアログ: await onSubmit を try/catch
                                    ├─ フィールドに紐付く → setError（項目下に赤字）
                                    └─ 紐付かない        → 上部Alert
                                  一覧操作(削除/Excel)の全体エラー → トースト
```

**ステータスの使い分け**：
- 422 = tsoa の入力検証（`details` にフィールド別メッセージ）
- 409 = UNIQUE制約違反（本PJでは username 重複）
- 404 = 対象なし / 500 = サーバーエラー

**関連ファイル**：
- `client/src/api/error.ts` … `ApiError` と `toFieldErrors`（422/409 → `{field: message}`）
- `client/src/api/accountAuth.ts` … `!res.ok` で `ApiError` を throw
- `components/accountAuth/AccountAuthFormDialog.tsx` … 失敗時 `setError` で項目下、紐付かないものは Alert
- `server/src/index.ts` … tsoa `ValidateError` を 422 JSON に整形

## 5. 出す場所を揃える（理想形・採用済み）

クライアント由来もサーバー由来も、**同じ場所に出す**とユーザーは由来を意識せず直せる。

| エラー | 出る場所 |
|---|---|
| 必須・形式（クライアント） | その項目の真下に赤字 |
| 重複・項目に紐付く検証（サーバー422/409） | **同じ項目の真下に赤字**（setError） |
| 通信断・500・項目に紐付かない | ダイアログ上部 Alert ／ 一覧操作はトースト |

例：username に対し「ユーザー名は必須」（クライアント）も「このユーザー名は既に使われています」（サーバー409）も**同じ username の下**に出る。

---

## 6. このアプリでの「どこまでやるか」

社内/客先の業務CRUDでは：
- **必須**：入力中の即時赤字（onTouched）、送信時の全体チェック、サーバーエラーの表示。
- **やりすぎ回避**：業務ルール（重複等）を全部フロントで再現しない → サーバーに任せ結果を出す。
- 安全はサーバーで担保。クライアントは親切さのため。

---

## 7. 世の中の一般解との対応

- クライアント：React Hook Form + Zod（主流ど真ん中）。
- サーバー：型/スキーマから検証生成（tsoa。FastAPI/Nestと同系譜）。
- エラー表示：フィールド別は項目へ、全体はトースト/バナー（標準）。
- 二重検証のDRY化：本PJは「**型はOpenAPIから生成して共有・検証はclient(Zod)/server(tsoa)で別**」。
  React＋将来PHPの構成では契約ファーストが正解（tRPC/共有Zodは TS-only 向け）。
