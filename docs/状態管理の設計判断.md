# 状態管理の設計判断（Redux Toolkit と TanStack Query）

「なぜ両方使うのか」「どちらが必然か」を説明できるようにするためのメモ。
（最終更新: 2026-06-30）

---

## 1. 大前提：状態には2種類あり、性質が正反対

| | サーバー状態 | クライアント状態 |
|---|---|---|
| 持ち主 | サーバー（手元はコピー） | 自分（ブラウザ） |
| 性質 | 非同期・古くなる・共有される | 同期的・常に正しい・自分専用 |
| 例（本PJ） | 車種一覧、account-authのレコード | ログイン状態、ダイアログ開閉、選択中ファイル名、処理A/Bの実行ステータス |

この2つは別問題。片方に最適な道具は、もう片方が苦手。

---

## 2. 役割分担

```
サーバーが知っているデータ   → TanStack Query
クライアントだけが知る状態   → Redux Toolkit
```

| 管理 | 対象（本PJ） |
|------|-------------|
| TanStack Query | 型式一覧・車種マスタ・各メンテテーブルのCRUD（`hooks/useAccountAuth.ts` 等） |
| Redux Toolkit | 認証(`authSlice`)、処理A/B実行状態(`jobsSlice`)、読込済みファイル名(`localFilesSlice`) |

判断の合言葉：
> 「サーバーが知っているデータ」は TanStack Query、「クライアントだけが知っている状態」は Redux。

---

## 3. なぜ TanStack Query なのか

サーバー状態は「取ってくる」だけでは済まない。次が必要になる：
キャッシュ / 重複排除 / ローディング・エラー / 再取得 / 更新後の無効化 / 競合制御。

これを Redux でやると、全スライスに `loading/error/data` の3点セットと thunk を手書きする
（昔ながらのボイラープレート）。TanStack Query はこれを標準で肩代わりする。

### 「sliceに保存して取り直せばいい」への回答

`保存する`のは簡単。難しいのは「いつ古い? 誰が取り直す? 重複は?」。そこを slice では全部自作する。

| やりたいこと | slice（自前） | TanStack Query |
|---|---|---|
| 複数画面が同時要求 | fetchが複数飛ぶ | 自動で1回にまとめる |
| 「N分は使い回し」 | タイマー自作 | `staleTime` 一行 |
| 追加後に一覧更新 | 手で再fetch dispatch | `invalidateQueries` で自動 |
| 条件別キャッシュ（型式をvehicleId別） | キー付きmap自作 | `queryKey:['katashiki', id]` |
| 古いfetchが新しい結果を上書き | 競合対策自作 | 順序制御・キャンセル込み |
| 画面復帰時に最新化 | 自分で | フォーカス時に自動 |

→ slice案は **TanStack Query を手作業で再発明する**ことになる。

```ts
// slice案（抜粋）：これを全テーブル分くり返す
state: { data: [], loading: false, error: null }
async function fetchVehicles() {
  dispatch(setLoading(true))
  try { dispatch(setData(await api.fetchVehicles())) }
  catch (e) { dispatch(setError(e)) }
  finally { dispatch(setLoading(false)) }
}

// TanStack Query：3行で同等＋キャッシュ・再取得・重複排除込み
useQuery({ queryKey: ['vehicles'], queryFn: fetchVehicles })
```

---

## 4. 一番効く再フレーム

> **Redux は「箱」。TanStack Query は「キャッシュの頭脳」。**
> Reduxは値を保管できるが、「いつ古いか・誰が取り直すか・重複をどうするか」は知らない。
> そこを自前で書くくらいなら、最初から専用ツールを使う方が速くて安全。

---

## 5. 「両方必要か / 必然か」── 濃淡をつけて答える

2つの必要度は同じではない。

- **TanStack Query → ほぼ必然**
  本PJは Express 越しの CRUD が主役 = サーバー状態が中心。
  使わない選択は「キャッシュ機構を自作する」ことを意味し、現実的でない。

- **Redux → 必然ではなく“正当化された選択”**
  画面横断の複雑なクライアント状態（特に処理A/Bの実行ステータス）があるから採用。
  もしそこが薄ければ Context / Zustand で代替も可能。

説明用の結論：
> TanStack Query は**サーバー状態のため**でアプリの形からして実質必須。
> Redux は**クライアント状態のため**で、画面横断の状態が複雑だから採用した選択（代替もありえる）。

---

## 6. slice案が正解になる場面

- 一度取ったら変わらない（起動時1回・以後不変）
- 重複もキャッシュも気にしなくてよい静的データ

→ こうしたものは slice や定数で十分。
頻繁に読み書きし更新後に最新化が要るもの（＝メンテ系テーブル全部）は TanStack Query が適。
