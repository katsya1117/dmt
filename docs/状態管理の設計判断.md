# 状態管理の設計判断（Redux Toolkit と RTK Query）

「なぜサーバー状態とクライアント状態を分けて考えるのか」「どちらが必然か」を説明できるようにするためのメモ。
（最終更新: 2026-07-14。TanStack QueryからRTK Queryへの切り替えを反映。切り替えの経緯は末尾「0. RTK Queryへの切り替え」参照）

---

## 0. RTK Queryへの切り替え（2026-07-14）

当初はTanStack Query（サーバー状態）+ Redux Toolkit（クライアント状態）の2ライブラリ構成で試験導入していたが、**RTK Query（Redux Toolkit同梱）へ統一**した。理由は「新しい技術だから」ではなく、以下の具体的な比較に基づく。

### 比較したポイント

| 観点 | TanStack Query | RTK Query |
|---|---|---|
| キャッシュ・invalidate・重複排除などの機能 | ○ | ○（ほぼ同等） |
| チームの既存知識との親和性 | 低い（新規学習） | 高い（Redux運用実績の延長） |
| 状態管理の思想 | サーバー状態/クライアント状態で**2つのパラダイム**が並走 | Reduxの1つのメンタルモデルに統一できる |
| 依存追加 | 新規ライブラリ | ゼロ（`@reduxjs/toolkit`に同梱） |
| APIの破壊的変更の頻度（15年運用の観点） | v1〜v5まで約6年で5回のメジャーバージョン。設定APIの変更もある | Redux本体のコアAPIは2015年からほぼ不変。RTK Queryも破壊的変更は少ない |
| バックエンド非依存性 | 高い（REST/GraphQL問わず） | RESTなら十分。本PJはtsoa/OpenAPIでREST契約が確定済みのため恩恵が薄い |

「バックエンド非依存性」を除き、機能面で必然性の差は無い。差が出るのは**15年という運用期間でのAPI安定性**と**チームの既存スキルとの一致度**で、いずれもRTK Query優位と判断した。

### 切り替えのタイミング

切り替え当時、TanStack Queryを使っていたのはaccount-auth機能とマスタ参照（車種/型式）の2箇所のみ（27画面中2画面）。影響範囲は`hooks/useAccountAuth.ts`・`hooks/useMaster.ts`・`lib/queryClient.ts`・`App.tsx`・Storybookのdecorator・`AccountAuthTable.tsx`の呼び出し方（`.mutate()`→トリガー関数呼び出し）程度で、量産フェーズ（残り25画面）に入る前の最小コストで完了した。この「新規プロジェクトは導入が最も安い」という理由づけは、後述7章でTanStack Query導入を正当化した際と同じロジックを、今回は「切り替え」に適用したもの。

### 実装上の要点

- **既存のAPI関数層（`api/*.ts`）は無変更**。RTK Queryは`store/services/*Api.ts`で`createApi` + `fakeBaseQuery` + `queryFn`により、既存の`api/*.ts`の関数（axios呼び出し・`toApiError`変換込み）をそのままラップするだけ。HTTP呼び出しやエラー変換のロジックを重複させない
- **エラー型（`ApiError`）はそのまま**。`queryFn`が`{ error: ApiError }`を返すことで、`.unwrap()`時に同じ`ApiError`インスタンスがthrowされる。`toFieldErrors()`等の呼び出し側は無変更
- **mutationの呼び方はRTK Query流に統一**（画面側も変更）。`create.mutate(...)`のようなオブジェクト形式ではなく、`const [create] = useCreateAccountAuth(); create(records).unwrap()`という「トリガー関数を直接呼ぶ」形にした。恒久的な互換シムは持たない方針（詳細は`docs/画面実装パターン.md`）
- **Storybookのdecorator**はストーリーごとに独立したRTK Queryキャッシュ（＝独立したReduxストア）を持たせるため、`store/index.ts`の`configureStore`をファクトリ関数`createStore()`として公開し、各ストーリーで新規呼び出しする形にした（TanStack Query版で`new QueryClient()`をストーリーごとに作っていたのと同じ意図）

## 1. 大前提：状態には2種類あり、性質が正反対

| | サーバー状態 | クライアント状態 |
|---|---|---|
| 持ち主 | サーバー（手元はコピー） | 自分（ブラウザ） |
| 性質 | 非同期・古くなる・共有される | 同期的・常に正しい・自分専用 |
| 例（本PJ） | 車種一覧、account-authのレコード | ログイン状態、ダイアログ開閉、選択中ファイル名、処理A/Bの実行ステータス |

この2つは別問題。片方に最適な道具は、もう片方が苦手。

### 見分ける質問（迷ったらこれ）

> **「ブラウザを閉じて別のPCで開き直したら、そのデータはまだ存在する？同じ？」**

- **YES**（まだある・同じ）→ **サーバー状態**（持ち主はサーバー、手元はコピー。古くなる）→ RTK Query
- **NO**（消える・その人だけ）→ **クライアント状態**（持ち主は自分のブラウザ。常に正しい）

言い換え：「**source of truth（本物）を誰が持つか**」。サーバーが持てばサーバー状態。

### つまずきポイント：クライアント状態 ≠ slice（状態は3バケツ）

RTK QueryもRTKのslice（`authSlice`等）も同じ「Redux Toolkit」の機能だが、役割は別物。

| 状態 | 持ち主 | 道具 |
|---|---|---|
| サーバー状態 | サーバー（手元はコピー） | **RTK Query**（`store/services/*Api.ts`） |
| クライアント状態（複数画面で共有） | 自分のブラウザ | **RTKのslice**（`store/slices/*.ts`） |
| クライアント状態（その画面だけ） | そのコンポーネント | **useState** |

「クライアント状態＝必ずslice」ではない。**大半は useState で十分**。sliceは画面横断で共有する時だけ。

### この案件の状態を仕分けると

| 状態 | 別PCで同じ？ | 分類 → 道具 |
|---|---|---|
| 車種一覧 / account-authレコード | 同じ | サーバー → RTK Query |
| ダイアログ開閉 / 選択中の行 | 消える | クライアント(局所) → useState |
| ログイン状態 | ※ | クライアント(共有) → slice |
| 処理A/B実行ステータス | ※※ | 今はクライアント(共有) → slice |
| 読込済みファイル名 | その人だけ | クライアント(共有) → slice |

- ※ ログインは紛らわしい：ユーザーの**情報そのもの**はサーバー状態。「**今このブラウザでログイン中か**」はクライアント状態。
- ※※ 処理A/Bは将来サーバーがジョブ状態を持ち**ポーリングで取る**なら**サーバー状態（RTK Query）に変わる**。現時点はクライアント保持でslice。

---

## 2. 役割分担

```
サーバーが知っているデータ   → RTK Query
クライアントだけが知る状態   → RTKのslice
```

どちらも同じReduxストア・同じ`@reduxjs/toolkit`パッケージの機能。ライブラリを2つ跨ぐ話ではなく、**1つのライブラリの中の2つの使い方**という位置づけ。

| 管理 | 対象（本PJ） |
|------|-------------|
| RTK Query | 型式一覧・車種マスタ・各メンテテーブルのCRUD（`store/services/*Api.ts`、`hooks/useAccountAuth.ts` 等） |
| RTKのslice | 認証(`authSlice`)、処理A/B実行状態(`jobsSlice`)、読込済みファイル名(`localFilesSlice`) |

判断の合言葉：
> 「サーバーが知っているデータ」は RTK Query、「クライアントだけが知っている状態」は slice。

---

## 3. なぜ RTK Query なのか

サーバー状態は「取ってくる」だけでは済まない。次が必要になる：
キャッシュ / 重複排除 / ローディング・エラー / 再取得 / 更新後の無効化 / 競合制御。

これを素のsliceでやると、全スライスに `loading/error/data` の3点セットと thunk を手書きする
（昔ながらのボイラープレート）。RTK Queryはこれを標準で肩代わりする。

### 「sliceに保存して取り直せばいい」への回答

`保存する`のは簡単。難しいのは「いつ古い? 誰が取り直す? 重複は?」。そこを素のsliceでは全部自作する。

| やりたいこと | slice（自前） | RTK Query |
|---|---|---|
| 複数画面が同時要求 | fetchが複数飛ぶ | 自動で1回にまとめる |
| 「N分は使い回し」 | タイマー自作 | `keepUnusedDataFor` 一行 |
| 追加後に一覧更新 | 手で再fetch dispatch | `invalidatesTags` で自動 |
| 条件別キャッシュ（型式をvehicleId別） | キー付きmap自作 | 引数ごとに自動でキャッシュが分かれる |
| 古いfetchが新しい結果を上書き | 競合対策自作 | 順序制御・キャンセル込み |
| 画面復帰時に最新化 | 自分で | `refetchOnFocus`設定で自動 |

→ slice案は **RTK Query を手作業で再発明する**ことになる。

```ts
// slice案（抜粋）：これを全テーブル分くり返す
state: { data: [], loading: false, error: null }
async function fetchVehicles() {
  dispatch(setLoading(true))
  try { dispatch(setData(await api.fetchVehicles())) }
  catch (e) { dispatch(setError(e)) }
  finally { dispatch(setLoading(false)) }
}

// RTK Query：createApiに1回書けば、あとは生成フックを呼ぶだけ
// （store/services/masterApi.ts）
vehicles: builder.query<Vehicle[], void>({
  queryFn: async () => { try { return { data: await fetchVehicles() } } catch (e) { return { error: e as Error } } },
})
// 呼び出し側（hooks/useMaster.ts）
masterApi.useVehiclesQuery() // data/isLoading/error込みで返る。キャッシュ・重複排除も自動
```

## 4. 一番効く再フレーム

> **RTKのsliceは「箱」。RTK Queryは「キャッシュの頭脳」。**
> sliceは値を保管できるが、「いつ古いか・誰が取り直すか・重複をどうするか」は知らない。
> そこを自前で書くくらいなら、同じRedux Toolkitに同梱の専用機能を使う方が速くて安全。

---

## 5. 「両方必要か / 必然か」── 濃淡をつけて答える

2つの必要度は同じではない。

- **RTK Query → ほぼ必然**
  本PJは Express 越しの CRUD が主役 = サーバー状態が中心。
  使わない選択は「キャッシュ機構を自作する」ことを意味し、現実的でない。

- **slice（クライアント状態の共有） → 必然ではなく“正当化された選択”**
  画面横断の複雑なクライアント状態（特に処理A/Bの実行ステータス）があるから採用。
  もしそこが薄ければ Context / Zustand で代替も可能。

説明用の結論：
> RTK Queryは**サーバー状態のため**でアプリの形からして実質必須。
> sliceは**クライアント状態のため**で、画面横断の状態が複雑だから採用した選択（代替もありえる）。

---

## 6. slice案が正解になる場面

- 一度取ったら変わらない（起動時1回・以後不変）
- 重複もキャッシュも気にしなくてよい静的データ

→ こうしたものは slice や定数で十分。
頻繁に読み書きし更新後に最新化が要るもの（＝メンテ系テーブル全部）は RTK Query が適。

## 7. 導入の正当化（説明材料）

サーバー状態管理ツール（RTK Query）を採用する理由を、人に説明するための材料。

### (1) 手書きの隠れコスト
データ取得画面は毎回 `useState(data/loading/error) + useEffect + 手動refetch` を書く。
これが**画面の数だけ重複**する。RTK Queryは生成フック1行で同等＋キャッシュ込み。
→「N画面 × 重複ロジックが消える」は定量的に示せる。

### (2) 防げる具体的バグ（最強の材料）
| バグ | 手書きだと | RTK Query |
|---|---|---|
| 更新したのに一覧が古いまま | refetch忘れ | `invalidatesTags`で自動 |
| 同じデータを多重リクエスト | 各所でfetch | 重複排除 |
| 古い応答が新しいデータを上書き | 競合対策を自作 | 順序制御・キャンセル込み |
| アンマウント後のsetState/leak | 自前guard | 自動キャンセル |
| ローディング/エラーUIの漏れ・不統一 | 画面ごとにバラつく | 標準で揃う |

> 「過去に実際に踏んだバグ」を1つ挙げ「これは構造的に消える」と言えると説得力が高い。

### (3) なぜ“今”か（急遽導入・切り替えへの返し）
- **新規（あるいは実装量が少ない今）は導入・切り替えが最も安い**。移行不要／最小コストでゼロから正しい型に乗せられる。後回しほど置換コストが積む
- 「前は入れ損ねた／別のツールを試験導入していた」は**今やる理由を補強**（同じ技術的負債を繰り返さない）
- 機能単位で段階導入でき大改修不要＝リスクは局所

### (4) なぜ他でなく RTK Query か（TanStack Queryとの比較）
- 機能面（キャッシュ・invalidate・重複排除）はTanStack Queryとほぼ同等。差は「業界シェア」ではなく**15年運用でのAPI安定性**と**チームの既存Redux知見との一致度**（詳細は0章）
- **新規依存ゼロ**：`@reduxjs/toolkit`に同梱されており、既にRedux Toolkitを採用している本PJでは追加の学習・依存管理コストが無い
- **状態管理が1つのメンタルモデルに統一される**：サーバー状態とクライアント状態を別々のライブラリの流儀で書かずに済む
- 本PJのバックエンドはtsoa/OpenAPIでREST契約が確定しているため、TanStack Query側の強み（バックエンド非依存）の恩恵は薄い

### (5) 反論への返し
| 言われそう | 返し |
|---|---|
| 学習コストが… | コアは生成される `useXxxQuery`/`useXxxMutation` の2種類。手書きキャッシュを学ぶより安く、既存のRedux知見の延長で読める |
| 依存が増える | 増えない。`@reduxjs/toolkit`に同梱（TanStack Query採用時と比べてむしろ依存が1つ減った） |
| sliceあるのに二重では？ | 役割が違う（サーバー状態 vs クライアント状態）が、**同じライブラリの機能**なので学習・保守コストの二重化は無い。本書1〜2章参照 |

### まとめの一言（プレゼン用）
> サーバー状態の管理は、放っておいても毎画面で手作りする羽目になる定型処理。
> 標準ツールに置き換えるだけで、再取得忘れ・競合・重複といった**バグのクラスごと消える**。
> 既にRedux Toolkitを採用している本PJでは、RTK Queryへの統一が最も追加コストの少ない選択でもある。
