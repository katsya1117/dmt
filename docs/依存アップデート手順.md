# 依存ライブラリ 安全アップデート手順書（普遍版）

依存を崩さずにライブラリを更新するための、プロジェクト非依存の手順。
「一気に全部上げて壊れ、原因が分からなくなる」を防ぐことが目的。

---

## 0. 大原則（これだけは守る）

1. **一度に1テーマだけ上げる。** 複数のメジャー更新を混ぜない。
2. **各更新のあとに必ず検証ゲートを通す**（型チェック→テスト→ビルド→実機）。
3. **1更新 = 1コミット。** 壊れたら `git revert` で即戻せる単位にする。
4. **`package.json` の `^` 制約は意図的な固定。** むやみに書き換えない。
5. **メジャー更新は CHANGELOG / 移行ガイドを読んでから。**

> 破ってよい例外はない。急ぐほど段階を踏む。

---

## 1. セマンティックバージョニング（semver）の読み方

`^1.4.2` の `1.4.2` は **メジャー.マイナー.パッチ**。

| 種類 | 例 | 意味 | 危険度 |
|------|-----|------|--------|
| パッチ | 1.4.2→1.4.3 | バグ修正のみ | 低。基本安全 |
| マイナー | 1.4.2→1.5.0 | 後方互換の機能追加 | 低〜中 |
| **メジャー** | 1.4.2→**2**.0.0 | **後方互換を壊す変更** | 高。要注意 |

- `^1.4.2` = `>=1.4.2 <2.0.0`（メジャーは上げない範囲）
- `~1.4.2` = `>=1.4.2 <1.5.0`（パッチのみ）

**`yarn upgrade` / `npm update`（引数なし）はこの `^`/`~` 範囲内でしか動かない**＝メジャーは上がらない＝比較的安全。
**メジャーを上げるのは `--latest`（yarn）/ `pkg@latest`（npm）を明示したときだけ。**

---

## 2. 事前準備（セーフティネットを張る）

更新を始める前に、戻れる状態を必ず作る。

```bash
# (1) 作業ツリーをクリーンにする（未コミットの変更を無くす）
git status            # クリーンであることを確認

# (2) 専用ブランチを切る
git switch -c deps/update-YYYYMMDD

# (3) 「更新前のグリーン」を確認しておく（ここが壊れてたら更新と無関係に失敗する）
#     ↓ プロジェクトの検証コマンドに置き換える
<型チェック>          # 例: npx tsc --noEmit
<テスト>              # 例: npm test / npx vitest run
<ビルド>              # 例: npm run build
```

→ **更新前にグリーンであることを確認**するのが肝。ここを飛ばすと「元から壊れてたのか、更新で壊したのか」が分からなくなる。

---

## 3. 現状把握（まだ何も変えない）

```bash
# 何がどれだけ遅れているか
npm outdated          # または yarn outdated

# 既知の脆弱性
npm audit             # または yarn audit
```

`outdated` の見方：
- **Current**（今）/ **Wanted**（^範囲内で上げられる先）/ **Latest**（最新）
- `Current < Wanted` … 範囲内更新で追従できる（安全枠）
- `Wanted < Latest` … メジャー跨ぎが必要（要注意枠）

ここで更新対象を**「安全枠」と「メジャー枠」に仕分け**する。

---

## 4. アップデートの実行順序

**安全なものから。崩れにくい順に積み上げる。**

### ステップ A：範囲内更新（パッチ/マイナー）をまとめて

```bash
npm update            # または yarn upgrade
```
→ 検証ゲート（第6章）→ 通れば **1コミット**。

これは `^` 範囲内なので基本安全。まずここで足場を固める。

### ステップ B：メジャー更新を「1つずつ」

関連パッケージは**グループ単位**で（例: `react` と `react-dom`、`@types/x` も一緒）。

```bash
# npm の場合
npm install react@latest react-dom@latest
npm install -D @types/react@latest @types/react-dom@latest

# yarn(v1) の場合（必ずパッケージ名を指定。引数なし --latest は厳禁）
yarn upgrade react react-dom --latest
```
→ 検証ゲート → 通れば **1コミット** → 次のメジャーへ。

**1グループ上げるたびにコミット。** これを繰り返す。

---

## 5. メジャー更新の前にやること

メジャーは「上げてみて祈る」をしない。先に読む。

1. **CHANGELOG / Migration Guide / Release Notes を開く**
   - breaking changes の節を必ず読む
   - 公式の **codemod（自動移行スクリプト）** があるか確認（React, Next, MUI 等は提供されることが多い）
2. **peer 依存の整合を確認する**
   - 例: テストランナー・バンドラ・プラグインは互いにバージョン要件を持つ
     （Vite ↔ Vitest ↔ プラグイン、Storybook ↔ Vite など）
   - 1つだけ突出して上げると peer 不整合で壊れる。**連動するものは一緒に上げる**
3. **インストール時の peer 警告を読む**
   - `ERESOLVE` / `peer dependency` 警告は無視しない。これが将来の不具合の予告

---

## 6. 検証ゲート（毎回・更新のたびに通す）

更新1回ごとに、この4段を上から順に。**1つでも赤なら次に進まない。**

```bash
# (1) 型チェック … 最も速く壊れを検知できる
<型チェック>          # 例: npx tsc --noEmit

# (2) テスト … 振る舞いの回帰を検知
<テスト>              # 例: npx vitest run

# (3) ビルド … 本番ビルドが通るか（dev では出ない問題が出る）
<ビルド>              # 例: npm run build

# (4) 実機 … 実際に起動して主要画面/主要操作を触る
<起動して動作確認>     # スクショ/手動。型もテストも素通りする実行時破壊を捕まえる
```

> (1)→(4) は「速くて軽い」→「遅いが本物」の順。早い段で落ちれば時間を節約できる。
> **(4) 実機を省かない。** 型もテストも通るのに実行時に白画面、はよくある。

---

## 7. 壊れたときのロールバック

検証ゲートで赤が出たら、**原因を直すより先に切り戻す**判断も有効。

```bash
# 直前の更新コミットだけ取り消す（履歴は残す）
git revert HEAD

# まだコミットしていない更新を捨てて直前の状態へ
git checkout -- package.json package-lock.json   # npm
git checkout -- package.json yarn.lock           # yarn
npm install   # または yarn install   ← lockに合わせて node_modules を戻す

# どうしても噛み合わない時は lock ごとクリーン再生成
rm -rf node_modules package-lock.json   # or yarn.lock
npm install
```

**1更新1コミットにしておけば、被害は必ず1コミット分に限定できる。** これが第0章のルールの効き目。

---

## 8. lockfile の扱い

- **lockfile（`package-lock.json` / `yarn.lock`）は必ずコミットする。** 環境間で同じ版を再現する要。
- **lockだけ変わって package.json が無変更**なら、それは範囲内のパッチ移動＝軽微。検証が通れば取り込んでよい。
- **複数のパッケージマネージャを混在させない。** 1つのディレクトリで npm と yarn を混ぜると lock が二重化して破綻する。
  - 例: あるディレクトリは `package-lock.json`（npm）、別は `yarn.lock`（yarn）と決まっているなら、各々その流儀で操作する。

## 9. npm / yarn コマンド対応表

| やりたいこと | npm | yarn v1 |
|---|---|---|
| 遅れの一覧 | `npm outdated` | `yarn outdated` |
| 脆弱性確認 | `npm audit` | `yarn audit` |
| high以上だけ判定 | `npm audit --audit-level=high` | `yarn audit --level high` |
| 範囲内で更新（安全） | `npm update` | `yarn upgrade` |
| 1個をメジャー更新 | `npm i pkg@latest` | `yarn upgrade pkg --latest` |
| 自動修正 | `npm audit fix` | （なし／手動） |
| lockに合わせ再現 | `npm ci` | `yarn install --frozen-lockfile` |

> yarn v3/v4(Berry) は `yarn npm audit` 等、コマンド体系が異なる。手元の `yarn -v` を先に確認。

---

## 10. やってはいけないこと（事故の典型）

- ❌ `yarn upgrade --latest`（**引数なし**）で全部一斉メジャー更新
- ❌ `npm audit fix --force` を中身を読まずに実行（依存を破壊的にダウングレードすることがある）
- ❌ 検証せずに複数メジャーを連続更新してまとめて1コミット
- ❌ peer 警告・`ERESOLVE` を無視
- ❌ lockfile をコミットしない／.gitignore する
- ❌ 作業ツリーが汚れたまま更新を始める（戻れなくなる）

## 11. 実行チェックリスト（コピペ用）

```
[ ] git status がクリーン
[ ] 専用ブランチを切った
[ ] 更新前の検証ゲートがグリーン
[ ] outdated / audit で対象を「安全枠」「メジャー枠」に仕分けた
[ ] ステップA: 範囲内更新 → 検証ゲート → コミット
[ ] ステップB: メジャーを1グループずつ
      [ ] CHANGELOG / 移行ガイド / codemod を確認
      [ ] peer 依存の連動を確認
      [ ] 更新 → 検証ゲート(型→テスト→ビルド→実機) → コミット
      [ ] 次のグループへ…
[ ] 最終 audit で high/critical が無いことを確認
[ ] lockfile をコミット
[ ] ブランチをマージ
```

## 補足：このリポジトリ固有の注意

- **client は yarn、server は npm** とパッケージマネージャが分かれている。
  各ディレクトリでその流儀（client=`yarn`, server=`npm`）を守ること。混ぜない。
- 検証コマンドの実体：
  - 型チェック: `npx tsc --noEmit`
  - テスト: `npx vitest run --project unit`
  - ビルド: `npm run build`（client）
  - 実機: ルートで `yarn dev` → `http://localhost:5173`
