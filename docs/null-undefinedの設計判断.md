# null と undefined の設計判断

型を設計するとき「ここは `?`（undefined）か `| null` か」を迷わないための指針。
（最終更新: 2026-07-01）

---

## 0. 一番深い定義

- **undefined** = 情報が無い（提供されていない／キーが存在しない）
- **null** = 情報はある。その内容が「空」だと**意図的に**言っている

> 覚え方： **undefined＝「知らない／渡されていない」、null＝「空だと分かっている」**
> （knowledge〔情報の有無〕 vs value〔値が空〕）
> 判断軸：「これは“その情報が無い”のか“情報として空っぽ”なのか?」

---

## 1. 核心：プロジェクトで1つ決めるものではない。「境界ごと」に決まる

null/undefined は層（境界）を跨ぐたびに、その境界の制約で変わる。DBはそのうちの1つを縛るだけ。

| 境界 | 使える表現 | 強制されること |
|---|---|---|
| DB(SQL) | NULL のみ | undefinedは無い。空は null 一択 |
| ワイヤ(JSON) | 値あり / null / **キー無し** | 3状態。「キー無し」がJSの undefined に対応。undefined自体はJSONに書けない |
| アプリ(TS) | null も undefined も両方 | ここは**自分で選ぶ**（自由が生まれる層） |
| UI(フォーム) | "" 空文字 / 未チェック 等 | また別の表現 |

→ 同じ「空」でも境界ごとに化ける。各ホップで「**写すのか・正規化するのか**」を決める。

---

## 2. 決定要因

1. **どの境界か** — DBは null強制、JSONは3状態、TSは自由。
2. **「不在」と「空」を区別する意味がドメインにあるか（最重要）**
   - 区別が要る典型＝**PATCH（部分更新）**：キー欠落=「触らない」、null=「空にする」→ 両方必須。
   - 区別が不要なら **どちらか1つに畳んでよい**（＝「うちは null 使わない」が成立する根拠）。
3. **シリアライズ挙動** — `JSON.stringify` は undefinedキーを消す／nullは残す。
   - 空を**載せたい**→ null ／ 項目ごと**消したい**→ undefined。
4. **相互運用** — SQL/PHP/Java 等は null しか持たない。統合境界では **null が共通語**。
5. **一貫性・安全（footgun回避）** — デフォルト引数・分割代入のデフォルトは undefined にしか効かない／`?.` は undefined を返す／`== null` は両方拾う。事故を減らすため1つに寄せることがある。

---

## 3. 「null を使わない」プロジェクトの正体

要因2で「不在と空を区別しない」、要因5で「無を1つに」と判断したチームは **undefined に寄せる**
（TS/JSのidiom・`?`/デフォルト引数が効く・JSONから消える）。
逆に **DB/他言語との往復が主役**なら、空が往復で生き残る **null に寄せる**。
好みではなく**境界構成への最適化**。

---

## 4. 実務ルール（迷ったらこれ）

- **読み取り型（レスポンス）**：全カラム常に存在 → null可は `x: T | null`（`?` は使わない）。
- **書き込み型（リクエスト）**：
  - フル送信（PUT的）なら読み取りと**対称**に `T | null`（`?` 不要）。
  - 真の部分更新（PATCH）を始めて初めて、専用Patch型で `?`（＝「触らない」）を導入。
- **optional(`?`) は「本当に省略しうる時」だけ**。「とりあえず任意」は禁物。
- 層をまたぐ変換は**境界（リポジトリ/API関数）で正規化**する（例: DBの tinyint ⇄ API の boolean も同じ発想）。

---

## 5. 本プロジェクトの落とし所

- レスポンスは全カラム常に存在（情報はある、値が空なだけ）→ **null**。
- フォームは全項目を毎回送る（不在が無い）→ **null か値**、undefined は出さない。
- 客先DB(MySQL)・PHP と往復する → **null が共通語**。

→ よって型は **`| null` で読み書き対称**、`?` は使わない。
（フロント内部だけで完結する一時状態は undefined 寄りが素直。層によって変わる点に注意。）

> 一言：null/undefined は“プロジェクトで1つ”ではなく“**境界ごと**”。
> 各境界の強制と「不在と空を区別する意味があるか」で決まる。区別不要なら1つに畳む。
