# 設計書メソッド 学習ノート

作成日: 2026-07-11。note.com「Webアプリの設計書を書いてみよう」（著者: 増田直実氏、全26回）を読み込んでの学習ノート。お寿司屋のWebアプリを題材に、ウォーターフォール型開発の各フェーズで作成すべき設計書を実例つきで解説しているシリーズ。

**検証について**: 初版（2026-07-11作成）は26本の記事を一括でサブエージェントに読ませて作成したところ、「落とし穴」欄などに**記事に無い内容が捏造されている**ことが判明した。そのため全26項目を1本ずつ原文と突き合わせて検証し直した（本版）。各項目の「目的」は基本的に記事からの引用または引用に近い要約。「読み手」「構成要素」「ポイント」は、**記事に明記されている場合のみ記載し、無い場合は「明記なし」と明記する**方針で統一した（一般的なソフトウェア工学の知識による補完はしていない）。

## このシリーズの構成（V字モデル）

各回のタイトルから読み取れる実際の工程は、単純な一直線ウォーターフォールではなく、**V字モデル**（前工程でテスト仕様を先に定義し、後工程でそれを実行する）になっている。

```
要求分析 ──────┐                              ┌── システムテスト実施(26)
  仕様分析 ────┼─ 方式設計 ─ 詳細設計 ─ 実装作業 ┼── 結合テスト実施(25)
                └(テスト仕様は前倒しで定義)      ┘
```

- No.4「システムテスト仕様書兼成績書」（要求分析フェーズ）→ 実行はNo.26。**これは確認済み**：No.26本文に「システムテスト仕様書兼成績書に基づいて、システムテストを実施します」と明記されている。
- No.11「結合テスト仕様書兼成績書」（仕様分析フェーズ）→ 実行はNo.25、という対応関係は**推定**。No.25本文にNo.11への明示的な参照は無いが、タイトル・テスト対象の一致から見て妥当な対応と考えられる。

## 1. 要求分析フェーズ

### ユースケース図（No.1）
- **目的**: 「システムにどのような機能があり、どのアクターがその機能を利用するのかを明確にするために作成します」
- **読み手**: 明記なし
- **構成要素**: アクター（お客様）、ユースケース（楕円）、`<<extend>>`（拡張関係）、`<<include>>`（包含関係）
- **ポイント**: 明記なし

### ユースケース記述（No.2）
- **目的**: 「ユースケース記述とは、各ユースケースの詳細を記述したもの」「アクターとシステムとのやり取りをユースケースごとに記述します」
- **読み手**: 明記なし
- **構成要素**: アクター、目的、事前条件、事後条件、正常処理、例外処理、関連（他のユースケースとの関連性）
- **ポイント**: 明記なし（各要素を明確に分離して記述している、という体裁面の特徴はあるが、明示的な「コツ」としての記述は無い）

### 分析レイアウトイメージ（No.3）
- **目的**: 「ユースケース記述の正常処理に書かれたアクターとシステムのやり取りを視覚化するために、簡単なレイアウトイメージを作成します」
- **読み手**: 明記なし
- **構成要素**: 明記なし（複数のレイアウトイメージ図が示されているのみ）
- **ポイント**: 明記なし

### システムテスト仕様書兼成績書（No.4）
- **目的**: 「システム構造が正しいかを確認するための仕様（システムテストの項目）を定めたもの」「実行結果が要求分析結果を満たしていることを確認するテスト」
- **読み手**: 明記なし
- **構成要素**: 明記なし（テスト項目は画像で提示されており、テキストでの列挙は無い）
- **位置づけ**: この時点では仕様定義のみ。実行（成績記入）はシリーズ最終回No.26で行われることが、No.26本文で明示的に確認できる

## 2. 仕様分析フェーズ

### データ型定義書（No.5）
- **目的**: 「業務で使用するデータを洗い出して、その仕様を記述したもの」
- **構成要素**: 番号・コード・区分・数量・金額・名称・日付といったデータ型の分類、商品データ（商品コード・商品名・単価）、顧客データ（顧客番号・ログイン名・パスワード）、購入データ（購入商品種類・購入商品数・配達先名・配達先住所）
- **読み手／ポイント**: 明記なし

### 画面遷移図（No.6）
- **目的**: 「各画面からどの画面に移動できるか、画面の遷移を図で表したもの」
- **構成要素**: 各画面、画面間の移動を示す矢印
- **読み手／ポイント**: 明記なし

### 画面仕様書（No.7）
- **目的**: 「要求分析作業で作成した、ユースケース記述、レイアウトイメージなどをもとに各画面の詳細な画面を設計します。画面で使用する項目（項目名）やボタンの構成、ユーザーの操作方法などを定義します。」
- **構成要素**: トップ画面、ログイン画面、ログイン後の遷移画面、商品画面、カート画面、購入画面、購入エラー画面、ログアウト画面、ログアウトエラー画面
- **読み手／ポイント**: 明記なし

### 分析パターン定義書（No.8）
クラス図版とシーケンス図版の2種類がある。
- **クラス図版の目的**: 「オブジェクト抽出のパターンに従って、構造のパターンを設定したもの」
- **シーケンス図版の目的**: 「オブジェクト抽出のパターンに従って、処理のパターンを設定したもの」
- **読み手／構成要素／ポイント**: 明記なし（図解中心で、テキストでの詳細な説明は無い）

### 分析シーケンス図（No.9）
- **目的**: 「各ユースケース記述（シナリオ）の基本的な処理を表す図」
- **読み手／構成要素／ポイント**: 明記なし（定義とシーケンス図が示されているのみ）

### 分析クラス図（No.10）
- **目的**: 「各ユースケース記述（シナリオ）で登場したオブジェクトごとに、処理を束ねてクラスを抽出した図」
- **読み手／構成要素／ポイント**: 明記なし

### 結合テスト仕様書兼成績書（No.11）
- **目的**: 「プログラム間の構造が正しいかを確認するための仕様（結合テストの項目）を定めたもの」「各ユースケースに関係するクラスおよび画面を結合し、機能性と画面遷移をテストします」
- **構成要素**: 商品を検索する／商品一覧を表示する／商品をカートに入れる／カートの商品を表示する／カートの商品を削除する／商品を購入する、各ユースケースのテスト項目
- **読み手／ポイント**: 明記なし
- **位置づけ**: 定義のみ。実行はNo.25と対応すると考えられる（前述の通り推定）

## 4. 詳細設計フェーズ

### 設計パターン定義書 クラス図（No.17）

- **読み手**: 明記なし

### 設計パターン定義書 シーケンス図（No.18）
- **目的**: 処理パターンごとに、仕様分析作業で作成した分析パターン定義書（シーケンス図）に、設計パターン定義書（クラス図）で追加されたクラスを加えて、処理のパターンを定義すること
- **構成要素**: 分析パターン定義書（シーケンス図）から引き継いだ内容、設計パターン定義書（クラス図）で追加されたクラス、メソッド名、引数
- **ポイント**: 「メソッド名や引数などは、可能な限り詳細に記述」する
- **読み手**: 明記なし

### 設計シーケンス図（No.19）
- **目的**: 各ユースケースの処理を、メソッド名・引数・戻り値まで具体的に記述する（実装に近い水準）
- **構成要素**: ユースケース番号（UC101等）、オブジェクト間の相互作用、具体的なメソッド呼び出しと引数、戻り値
- **読み手／ポイント**: 明記なし

### 設計クラス図（No.20）
- **目的**: 「設計シーケンス図ごとに登場したオブジェクトを束ねてクラスを抽出した図」「クラスの構造（属性、操作、他クラスとの関係）を記述」するもの
- **構成要素**: UC101ログインする、UC102ログアウトする、UC201〜UC206（検索・一覧表示・カート関連機能）ごとの設計クラス図
- **読み手／ポイント**: 明記なし

### クラス定義書（No.21）
- **目的**: 「クラスに必要な属性や操作および可視性などを、わかりやすくドキュメントとして残す」
- **読み手／構成要素／ポイント**: 明記なし（目的文中に属性・操作・可視性への言及はあるが、構成要素としての列挙やコツの明記は無い）

### メソッド仕様書（No.22）
- **目的**: 「メソッド仕様書を記述することで、実装前にロジックを明らかにできます。」
- **構成要素**: CustomerDAO・ProductDAO・PurchaseDAOの3クラスについてのメソッド仕様（詳細は画像で提示）
- **読み手／ポイント**: 明記なし

### クラスカテゴリテスト仕様書兼成績書（No.23）
- **目的**: 個々のクラス単位でのホワイトボックステスト。テスト項目の仕様と結果記録を兼ねる

- **読み手**: 明記なし

## このプロジェクトへの適用にあたって

（2026-07-14更新：着手後の進捗を反映）

d社マニュアルメンテナンスツール（React+TypeScriptクライアント／Express(BFF)／将来的に客先PHPと連携）は、`docs/design/`配下にアカウント認証機能を題材にしたパイロットとして、下記の通りJavaのDAO/クラス設計に強く紐づく工程（分析・設計それぞれのクラス図/シーケンス図の2重化、クラス定義書・メソッド仕様書の細分）は本プロジェクトの技術スタック（TypeScriptの型がコード上の正典、hooksベースで継承関係を持つクラス設計を行わない）に合わせて統合・省略し、実装済み。

### まだ無いもの（次にやるなら）

- **認証・認可設計書**: 社外ユーザー（顧客社員）のログイン方式・ロール別権限マトリクス。CLAUDE.md・`デプロイ構成.md`で必要性は明記されているが、専用の設計書はまだ無い
- **セキュリティ設計書**: ファイルアップロードのバリデーション、CORS、レート制限など
- **非機能要件書**: 想定同時接続数・レスポンス目標・可用性・バックアップ方針
- **account-authパイロットの他テーブルへの横展開**: 現状account-auth一機能のみが設計書化されている。他のメンテ系テーブル（車種・お知らせ情報等）は`docs/画面実装パターン.md`の型をなぞって同様に作成する想定
