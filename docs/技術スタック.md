# 技術スタック（クライアント）

d社マニュアルメンテナンスツールのフロントエンド技術選定と、その採用理由・使い方の指針。
アーキテクチャ・レイヤ構成の正式な定義は[design/07_アプリケーション方式設計書](design/07_アプリケーション方式設計書.md)を参照（本書はライブラリ選定理由に絞る）。
（最終更新: 2026-07-14）

## 全体像

社内・運用PC常駐の業務ツール。27画面の大半はマスタテーブルのCRUDとファイルアップロード。
ReactはExpressとだけ通信し、PHP・処理A/B・顧客DBを直接意識しない（CLAUDE.md の設計方針に従う）。

## 採用ライブラリ一覧

| 分類 | ライブラリ | バージョン | 役割・採用理由 |
|------|-----------|-----------|----------------|
| 基盤 | React + TypeScript | React 19 / TS 6 | 業務SPAの標準 |
| ビルド | Vite | 8 | 高速な開発サーバー・ビルド |
| ルーティング | React Router | 7.18 | ライブラリモード（BrowserRouter）で使用。型は本体同梱（`@types/react-router-dom`は不要） |
| UI | MUI (Material UI) | 9.1 | 管理画面・密なテーブル表示・フォームに最も向く。CRUD中心の本ツールに最適 |
| スタイル | emotion | 11 | MUIの内部依存。Tailwindは共存コストを嫌い不採用（一度入れたが撤去） |
| 状態管理 | Redux Toolkit + react-redux | RTK 2.12 | クライアント状態（ログイン・処理A/Bの実行ステータス・読込済みファイル名）とサーバー状態（RTK Query。Expressから取得するデータのキャッシュ・取得管理）を単一ストアに統一。2026-07-14、TanStack Queryから移行（理由は[状態管理の設計判断](状態管理の設計判断.md)） |
| フォーム | React Hook Form + Zod + @hookform/resolvers | RHF 7.80 / Zod 4.4 | CRUDフォームの入力管理とバリデーション。スキーマから型もバリデーションも一元化 |
| HTTP通信 | axios | 1.18 | 共通インスタンス（`api/http.ts`）＋エラー変換（`toApiError`）を一元化するため、組み込みfetchから移行（2026-07-10） |
| APIモック | MSW | 2.14 | Storybook・Vitest両方でAPIをモック。実サーバーなしで開発・テスト可 |
| テスト | Vitest + Testing Library | Vitest 4.1 | ユニット（hooks/純粋関数）とコンポーネントテスト |
| カタログ | Storybook | 10.4 | コンポーネント駆動開発の練習・仕様の可視化（+ Playwrightブラウザテスト） |
| Excel解析 | exceljs | 4.4 | Excel取り込みのパース。旧xlsx(SheetJS)はnpm版が塩漬け＋CVEのため置換 |

> サーバー側: Express 5 / better-sqlite3 12 / **TypeScript 6** / 組み込みfetch採用（node-fetch廃止）。
> ※ tsoa 6.6 は peer 上 TS5系要求だが、実運用では TS6 で生成・型チェック・実行とも問題なく動作（peer警告は無視）。
> API層は **tsoa**（型からルート・入力検証・OpenAPI仕様を自動生成）＋ **swagger-ui-express**（/api-docs）。
> クライアント型は **openapi-typescript** で仕様から生成（`yarn gen:api`）＝client/serverの型を単一の真実に。

## 状態管理の役割分担

> 「サーバーが知っているデータ」は RTK Query、「クライアントだけが知っている状態」は RTKのslice。どちらも同じReduxストアに乗る。
> **RTK Queryを選んだ理由（TanStack Queryとの比較）は [状態管理の設計判断](状態管理の設計判断.md) に詳述。**

| 状態 | 管理 |
|------|------|
| ログイン・権限 | RTK slice (`authSlice`) |
| 処理A/B・アップロードの実行状態 | RTK slice (`jobsSlice`) |
| 読み込み済みローカルファイル名 | RTK slice (`localFilesSlice`) |
| 型式一覧・ファイル一覧・マスタテーブル | RTK Query (`store/services/*Api.ts`) |

必要・不必要は作りながら判断する方針（最初から作り込まない）。

## ディレクトリ構成

```
client/src/
  api/          # API呼び出し関数（fetch）と型定義
  hooks/        # カスタムフック（+ .test.ts を併置）
  store/        # Redux store。slices/（クライアント状態）と services/（RTK Query APIスライス）
  mocks/        # MSW（handlers / data / browser / server / vitest-setup）
  components/
    form/       # RhfTextField など RHF+MUI 連携の汎用フォーム部品
  pages/        # 27画面のコンポーネント
```

## テスト方針（テスタビリティ最重視）

完成後にテストを書くのは辛いので、構造で担保する：

1. **API アクセスはフックに閉じ込める** — コンポーネントは `useXxx()` だけ知る。テスト時はMSWで差し替え
2. **ビジネスロジックは純粋関数に出す** — Vitestで普通の関数テスト
3. **MSW でAPIをモック** — Storybook・Vitest共用（`src/mocks/`）
4. **Storybook のストーリー＝仕様** — 実装と同時に書く

実行コマンド:
- `yarn dev` — アプリ (http://localhost:5173)
- `yarn storybook` — Storybook (http://localhost:6006)
- `npx vitest run --project unit` — ユニットテスト
- パッケージ管理は **yarn** を使用

## CSS方針

スタイルの責務は **MUIのみ**（`sx` prop + `src/theme.ts`）。
MUI感を抑えるため theme.ts で影を薄く・角丸控えめ・カラーをニュートラルに・日本語フォント優先。

## 採用済み：一覧テーブルは MUI X DataGrid（Community・無料）

2026-07-13、アカウント認証機能で採用。差分プレビューダイアログが大量データ（実測20000行）で約13.7秒ブラウザを固まらせる問題を受け、仮想化目的で導入した。Community版（無料・MIT）に行の仮想化が標準搭載されており、Pro/Premium限定機能ではない（公式ドキュメントで確認済み）。導入後の実測: 20000行のプレビュー表示が約1秒、メインスレッド最大ブロックは0.3秒程度に短縮（詳細は`docs/アカウント認証_Excel取り込み設計.md`）。

15年運用を見据え、自作の仮想化（TanStack Virtual等）より保守負担が小さい既製品を選んだ。今後、大量データを扱う他画面（型式選択・約700件等）もこのDataGridパターンを踏襲する（`docs/画面実装パターン.md`参照）。
