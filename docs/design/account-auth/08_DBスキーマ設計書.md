# DBスキーマ設計書 — アカウント認証機能

（作成: 2026-07-13。フィールドレベルの型定義は`05_データ型定義書.md`を参照。本書は格納方式・設計判断を扱う）

## 目的

アカウント認証情報（得意先の販売会社・店舗ごとのログインアカウント）を、どのように格納するかを設計する。

## 対象データ

`account_auth`テーブル1つ（現時点で関連テーブルは無い。`09_テーブル関連図.md`参照）。

## 格納方式に関する設計判断

### 1. デモ環境と本番環境でDB実体が異なる（B3方式）

| 環境 | 実体 | 備考 |
|---|---|---|
| デモ・開発（現状） | SQLite（`server/data/demo.db`） | `server/src/db.ts`で定義。客先MySQLと**フィールド構成を一致**させている |
| 本番 | 客先MySQL（`account_auth`テーブル） | 既存の客先システムが使用している実テーブル |

客先DBに未接続の開発段階のため、SQLiteを「客先DBの代役」として使う（プロジェクト内でB3方式と呼称）。**差し替え点は`server/src/repositories/accountAuth.ts`のみ**とし、コントローラ・API関数・フック・画面は本番移行時も無変更で済む設計にしている（`07_アプリケーション方式設計書.md`のレイヤ構成に基づく）。

### 2. 論理削除方式（物理削除はしない）

`delfg`（tinyint(1)）で論理削除を表現する。客先DBで物理削除が不調という制約があるため、DELETE操作自体をアプリ側でも無効化している（`server/src/controllers/accountAuthController.ts`のDELETEルートはコメントアウト）。

削除・復元の設計は以下の安全ルールに基づく（詳細: `docs/アカウント認証_Excel取り込み設計.md`）:
- 「ファイルに無い＝削除」にはしない。削除は明示的に`delfg=1`と指定された行のみ
- リストアは`delfg`を`0`に戻すだけの単純な操作（別テーブルへの退避等はしない）

### 3. 一意性の担保

`username`にUNIQUE制約（DBレベル）。アプリ側リポジトリ層（`createAccountAuth`）でもINSERT時にDBの制約違反を捕捉し、409エラーとしてクライアントに伝える（`docs/バリデーションとエラー処理.md`参照）。

### 4. 文字コード・照合順序（本番MySQL）

`CHARACTER SET utf8`・`COLLATE utf8_bin`。`utf8_bin`はバイナリ照合＝**大文字小文字を区別する**比較になる。`username`や`password`の一致判定に影響するため、本番接続時の検索・重複判定ロジックがこの前提に合っているか要確認（現状のSQLiteデモは既定の照合順序で動いており、この区別を厳密に再現できていない可能性がある）。

### 5. タイムスタンプの扱い

- `reg_date`（登録日時）: アプリ側が明示的にセットする（DBの自動生成に依存しない）
- `upd_date`（更新日時）: 本番MySQLでは`ON UPDATE CURRENT_TIMESTAMP`でDBが自動更新するが、**SQLiteデモ側は自動更新の仕組みが無いため、アプリのリポジトリ層で明示的に現在時刻をセットしている**（`server/src/repositories/accountAuth.ts`の`nowStr()`）。本番移行時、DB側の自動更新とアプリ側のセットが二重に走っても実害は無いが、時刻のズレ（DBサーバー時刻 vs アプリサーバー時刻）が起きうる点は留意

## 未確定・要検討事項

- 本番MySQL接続時、`utf8_bin`の大文字小文字区別がアプリの想定と一致しているか
- パスワードの平文格納について、セキュリティ要件上問題ないか（既存客先システムの仕様に準拠していると推測されるが未確認。`05_データ型定義書.md`にも記載）
- 将来的にテーブルが増えた場合の命名規則（現状`account_auth`のようなsnake_case、単数系接尾辞`_auth`等の明確なルールはまだ無い）
