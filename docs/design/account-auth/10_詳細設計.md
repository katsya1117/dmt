## 1. API契約（tsoaコントローラより）

### アカウント認証CRUD（`server/src/controllers/accountAuthController.ts`）

| メソッド | パス | リクエスト | レスポンス | ステータス |
|---|---|---|---|---|
| GET | `/api/account-auth` | クエリ`includeDeleted?: boolean` | `AccountAuth[]` | 200 |
| POST | `/api/account-auth` | `{ records: AccountAuthInput[] }` | `{ inserted: number }` または `{ error }` | 201 / 409 |
| PUT | `/api/account-auth/{id}` | `AccountAuthInput` | `AccountAuth` または `{ error }` | 200 / 404 / 409 |
| ~~DELETE~~ | ~~`/api/account-auth/{id}`~~ | - | - | **未開放**（コントローラ内でコメントアウト） |

### Excel取り込み（`server/src/controllers/accountAuthImportController.ts`）

| メソッド | パス | リクエスト | レスポンス | ステータス |
|---|---|---|---|---|
| POST | `/api/account-auth/import/preview` | `multipart/form-data`（フィールド名`file`） | `ImportDiff`（`added`/`changed`/`deleted`/`restored`/`unchangedCount`/`skippedDuplicateUsernames`） | 200 |
| POST | `/api/account-auth/import/apply` | 同上 | `ApplyImportResult`（`inserted`/`updated`/`deleted`/`restored`） または `{ error, errors? }` | 200 / 400 |

`ImportDiff`・`ApplyImportResult`の型定義、差分判定ロジックの詳細は`docs/アカウント認証_Excel取り込み設計.md`を参照。

| 関数 | 役割 |
|---|---|
| `listAccountAuth()` | 削除済みを除いた一覧取得 |
| `listAllAccountAuth()` | 削除済み含む全件取得（差分判定・リストア表示用） |

| `updateAccountAuth(id, input)` | 更新。`delfg`の値に関わらず常に更新可能（2026-07-08修正：旧実装は`delfg=0`を条件にしており削除済み行を復元できないバグがあった） |
| `applyAccountAuthImport(params)` | Excel差分の一括反映。1トランザクションでadded/changed/deleted/restoredを処理 |

## 6. 検索・絞り込みが速い仕組み（`AccountAuthTable.tsx`）

No./ユーザー名の検索欄（`searchText`）は、入力のたびにサーバーへ問い合わせず、以下3点の組み合わせで即座に結果が反映される。

1. **通信が発生しない**: `useAccountAuthList(true)`で取得済みの全件（`data`。削除済み含む）は、TanStack Queryのキャッシュとして既にブラウザのメモリ上にある。検索欄の`onChange`は、この`data`に対して`Array.filter()`を実行するだけで、サーバーへの再問い合わせは行わない
   ```ts
   const filteredData = data?.filter((row) => {
     const q = searchText.trim().toLowerCase()
     if (!q) return true
     return String(row.number ?? '').includes(q) || row.username.toLowerCase().includes(q)
   })
   ```
2. **フィルタ処理自体が軽い**: `number`と`username`に対する単純な文字列包含判定を全行に対して行う線形スキャン。数万件規模でも数ミリ秒未満で完了する
3. **DataGridの仮想化が描画コストを結果件数から切り離す**: `filteredData`をそのまま`DataGrid`の`rows`に渡しているが、実際にDOMへ描画されるのはスクロール中に画面へ入っている行だけ。絞り込み結果が1件でも数千件でも、描画の重さはほぼ一定になる

**注意点**: `filteredData`はレンダーのたびに新しい配列として再計算される（メモ化していない）。現状の規模（〜数万行）では問題にならないが、将来さらにデータ量が増えて検索のたびに体感の遅延が出るようであれば、`useMemo`でのメモ化やデバウンス処理の追加を検討する。
