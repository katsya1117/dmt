# シーケンス図 — アカウント認証機能

（作成: 2026-07-13。`AccountAuthTable.tsx`・`AccountAuthFormDialog.tsx`・`useAccountAuth.ts`・`accountAuthController.ts`・`accountAuthImportController.ts`・`repositories/accountAuth.ts`の実装を基に作成。参加者名は`docs/画面実装パターン.md`の記法に合わせる）

> 以下の`DB`参加者は現状（デモ）のSQLiteを指す。本番ではリポジトリの実装が客先PHPへのAPI呼び出しに置き換わり、DBを直接操作するのは客先PHPのみになる（Expressから直接のSQL発行は無い）。

## 1. UC-A02: 新規追加

```mermaid
sequenceDiagram
  participant U as 利用者
  participant D as フォームダイアログ(AccountAuthFormDialog)
  participant P as 一覧画面(AccountAuthTable)
  participant H as フック(useAccountAuthMutations)
  participant A as API関数(createAccountAuth)
  participant C as コントローラ(accountAuthController)
  participant Repo as リポジトリ
  participant DB as DB

  U->>P: 「新規追加」ボタンを押す
  P->>D: ダイアログを開く（target=null）
  U->>D: 項目を入力し「追加」を押す
  D->>D: クライアント検証（Zod）
  alt 検証NG
    D-->>U: 該当項目に赤字エラー（送信しない）
  else 検証OK
    D->>P: onSubmit(input)
    P->>H: create.mutateAsync([input])
    H->>A: createAccountAuth(records)
    A->>C: POST /api/account-auth
    C->>Repo: createAccountAuth(records)
    Repo->>DB: INSERT INTO account_auth
    alt UNIQUE制約違反
      DB-->>Repo: エラー
      Repo-->>C: throw
      C-->>A: 409 { error }
      A-->>H: throw ApiError(409)
      H-->>D: reject
      D-->>U: username欄に「このユーザー名は既に使われています」
    else 成功
      DB-->>Repo: OK
      Repo-->>C: { inserted: 1 }
      C-->>A: 201 { inserted: 1 }
      A-->>H: { inserted: 1 }
      H->>H: invalidateQueries（一覧のキャッシュ無効化）
      H-->>P: onSuccess
      P->>P: ダイアログを閉じ、トースト表示
      P->>H: 一覧を自動再取得
    end
  end
```

## 2. UC-A03/04/05: 編集（削除・リストアを含む）

```mermaid
sequenceDiagram
  participant U as 利用者
  participant D as フォームダイアログ
  participant P as 一覧画面
  participant H as フック
  participant A as API関数(updateAccountAuth)
  participant C as コントローラ
  participant Repo as リポジトリ
  participant DB as DB

  U->>P: 対象行の「編集」アイコンを押す

  D->>P: onSubmit(input)
  P->>H: update.mutateAsync({ id, input })
  H->>A: updateAccountAuth(id, input)
  A->>C: PUT /api/account-auth/{id}
  C->>Repo: updateAccountAuth(id, input)
  Repo->>DB: UPDATE account_auth SET ... WHERE id=?
  Note over Repo,DB: delfgの現在値に関わらず更新可能<br/>（2026-07-08修正：旧実装はdelfg=0が条件でリストア不可だった）
  DB-->>Repo: OK
  Repo-->>C: 更新後の行
  C-->>A: 200 AccountAuth
  A-->>H: AccountAuth
  H->>H: invalidateQueries
  H-->>P: onSuccess
  P->>P: ダイアログを閉じ、トースト表示（状態列も更新される）
```

## 3. UC-A06/07: Excel取り込み（差分プレビュー→適用）

```mermaid
sequenceDiagram
  participant U as 利用者
  participant P as 一覧画面
  participant A as API関数
  participant C as import用コントローラ
  participant Parse as parseAccountAuthExcelBuffer
  participant Diff as computeImportDiff
  participant Repo as リポジトリ
  participant DB as DB
  participant Dlg as 差分プレビューダイアログ

  U->>P: 「Excel取り込み」ボタン→ファイル選択
  P->>A: previewImport(file)
  A->>C: POST /api/account-auth/import/preview（multipart）
  C->>Parse: parseAccountAuthExcelBuffer(file.buffer)
  Parse-->>C: AccountAuthInput[]
  C->>Repo: listAllAccountAuth()（削除済み含む全件）
  Repo->>DB: SELECT *
  DB-->>Repo: 全行
  Repo-->>C: AccountAuth[]
  C->>Diff: computeImportDiff(records, current)
  Diff-->>C: ImportDiff（added/changed/deleted/restored/unchangedCount）
  C-->>A: 200 ImportDiff
  A-->>P: ImportDiff
  P->>Dlg: ダイアログを開き差分をテーブル表示

  U->>Dlg: 内容確認後「この内容で適用」を押す
  opt 変更件数が50件以上
    Dlg-->>U: ブラウザの確認ダイアログ（confirm）
  end
  Dlg->>P: 適用実行
  P->>A: applyImport(file)
  A->>C: POST /api/account-auth/import/apply（multipart・同じファイルを再送）
  C->>Parse: parseAccountAuthExcelBuffer(file.buffer)
  Parse-->>C: AccountAuthInput[]
  C->>C: validateImportRecords(records)
  alt 検証NG（username空・重複）
    C-->>A: 400 { error, errors }
    A-->>P: throw
    P-->>U: エラートースト（ダイアログは閉じない）
  else 検証OK
    C->>Repo: listAllAccountAuth()（再読込。プレビュー後のDB変化に追従）
    Repo->>DB: SELECT *
    DB-->>Repo: 全行
    C->>Diff: computeImportDiff(records, current)（再計算）
    Diff-->>C: ImportDiff
    C->>Repo: applyAccountAuthImport(diffを変換したパラメータ)
    Repo->>DB: 1トランザクションでINSERT/UPDATE（added/changed/deleted/restored）
    DB-->>Repo: OK
    Repo-->>C: ApplyImportResult
    C-->>A: 200 ApplyImportResult
    A-->>P: ApplyImportResult
    P->>Dlg: ダイアログを閉じる
    P->>P: トースト表示（件数内訳）、一覧を再取得
  end
```

## 補足

- 差分プレビュー〜適用の間にDBが変化していても安全なように、**適用時に差分を再計算**している（プレビュー結果をそのまま信用しない）
- パースはサーバー側で行う（`parseAccountAuthExcelBuffer`）。クライアントはファイルをそのままアップロードするだけ（2026-07-10の設計変更。理由は`docs/アカウント認証_Excel取り込み設計.md`参照）
