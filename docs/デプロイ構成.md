# デプロイ構成・ハード構成（見積もり用）

（最終更新: 2026-07-14）

**正式な構成図・拠点間通信の定義は[design/06_システム構成図](design/06_システム構成図.md)**。本書はそれを踏まえた**見積もり観点の実装寄りの詳細**（コスト要因・運用PCのディレクトリ構成・ブラウザのローカルファイル制約という設計背景）に絞る。拠点ごとの役割・通信経路の記述は06と矛盾する場合06を正とする。

本ツールは **社内** と **客先** の2拠点にデプロイする。同一のReact/Expressコードベースを、
有効にする機能を**権限（ロール）で出し分け**て両拠点に配置する。
React ⇄ Express が HTTP で疎結合なため、この「置き場所と権限で出し分け」が無理なく成立する。

## 全体図

```
■ 社内（運用拠点）
┌──────────────────────────────────────┐
│ 運用PC（Windows）★1台で兼任             │
│   ・操作担当者のブラウザ（操作端末）      │
│   ・Express（全機能）                    │
│   ・処理A（VB）/ 処理B（Adobe）          │
│   ・マスタ/型式別フォルダ（ファイル保管）  │
│   → 社内スタッフが使用                   │
└──────────────────────────────────────┘
            │ リソース・型式別DBをアップロード
            │ （Express → 顧客PHP へ HTTP multipart POST）
            ▼
■ 客先
┌──────────────────────────────────────┐
│ 顧客サーバー環境（既存）                  │
│   ・顧客PHPサーバー（HTMLマニュアル配信）  │
│   ・電子マニュアルDB（MySQL）/ 共通DB     │
│   ・Express（機能を絞った版）★最低限設置   │
│   → 顧客の社員が DBメンテ等で使用         │
└──────────────────────────────────────┘
```

## 拠点ごとの役割

| | 社内（運用PC） | 客先 |
|---|---|---|
| OS | **Windows必須**（処理A/B） | Linux可（軽い） |
| 主な機能 | ファイル配置・処理A/B・アップロード（全機能） | DBメンテ等のデータ操作のみ |
| 利用者 | 社内スタッフ | 顧客の社員（社外ユーザー） |
| 触るデータ | マスタファイル・型式別フォルダ | 顧客側DB（電子マニュアルDB等。**必ず顧客PHP経由**、Expressから直接接続はしない） |
| 新規ハード | Windows機 1台 | **不要な可能性大**（既存顧客サーバーにNode＋アプリ設置） |

## なぜこの分け方か

- **処理A/B（VB・Adobe）はWindows必須** → 社内に集約。ファイルも処理も利用者も社内で完結
- **客先機能はデータ操作のみ**でWindows/ファイル不要 → 顧客PHP（DBを直接操作できる唯一の存在）の隣に軽量Expressを置き、PHP経由でDBを操作するのが自然
- 中央集約（社内Expressに全員が繋ぐ）案は、社内運用PCをインターネット露出させる不自然な形になるため不採用寄り（いずれにせよExpressがDBへ直接アクセスすることは無い）

## ブラウザのローカルファイル制約（重要な前提）

- ブラウザはローカルファイルに自由アクセスできない（セキュリティ制約）。AIR→Web化の最大トレードオフ
- 解決策：ファイル操作はExpressが肩代わり（サーバー側 fs でフォルダを読む）
- 運用フロー：担当者が**事前に**運用PCのフォルダへファイルを配置 → ブラウザで操作 → Expressが読む
- → Expressはファイルが見える運用PCに同居が必然。複数PCからのアクセスもExpress経由なので問題なし

## 客先利用が必須化させる要件

顧客の社員（社外ユーザー）が使うため、以下が中核機能に格上げ：

- **認証・権限管理**（画面: 権限管理／ユーザー権限管理／D3セキュリティ認証）
- ロールによる機能の出し分け（社内=全機能／顧客社員=DBメンテのみ 等）
- 客先Expressのネットワーク露出範囲の設計（顧客LAN内に閉じる等）

## 見積もりのコスト要因

- **社内**: Windows OSライセンス（Server版推奨）/ Adobeライセンス / マシンスペック（PDF・SWF変換は重め）/ 顧客サーバーへの通信経路（VPN等）/ Expressのサービス化（NSSM等）構築工数
- **客先**: 原則ハード費なし。Node＋アプリの**設置作業工数**。顧客がNode導入を許可するかは要確認

## 未確認・要確認事項

- 顧客サーバーにNode（Express）を載せられるか（顧客の運用ポリシー）
- 客先Expressのネットワーク到達範囲（顧客LAN内／外部公開の有無）
- 社内→顧客サーバーへのアップロード経路（VPN/専用線/インターネット）

## 運用PC（Windows）のディレクトリ構成イメージ

マスタXML → 処理A → 処理B → アップロード、の流れに沿ったフォルダ配置。

```
C:\manual-tool\                          ← ツール一式の親
│
├─ app\                                  ← このWebアプリ本体
│   ├─ server\                           ← Express（常駐サービス）
│   │   ├─ src\ または dist\
│   │   └─ .env                          ← パス設定（下記フォルダを指す）
│   └─ client\                           ← ビルド済みReact（dist）をExpressが配信
│
├─ processes\                            ← 別処理の実行体
│   ├─ process-a\                        ← VB製。XML分割＋型式別DB生成
│   └─ process-b\                        ← Python＋Adobe。PDF/SWF変換
│
└─ data\                                 ← ★ファイルの実体。処理の入出力（現 mock/samba 相当）
    │
    ├─ master\                           ← ① マスタXML置き場（担当者が手で配置）
    │   └─ <車種>\<型式>\*.xml
    │
    ├─ work\                             ← 処理A/Bの中間生成（分割XML・HTML等）
    │   └─ <車種>\<型式>\...
    │
    └─ upload\                           ← ② アップロード対象（処理B出力＋画像）
        └─ <型式>\
            ├─ *.xml                     ← 分割済みXML
            ├─ *.pdf / *.swf             ← マニュアル画像化
            ├─ *.svg / *.swf             ← 図版データ
            ├─ caption.xml               ← キャプション
            └─ katashiki.db              ← 型式別DB（処理Aが生成）
```

### データの流れ

```
master\（XML配置：担当者が手で置く入口）
   │  処理A：XML分割 ＋ 型式別DB生成
   ▼
work\（中間：分割XML・HTML。ツールは基本触らない作業場）
   │  処理B：変更分をソート → PDF/SWF変換
   ▼
upload\（型式別フォルダ：処理の出口）── Express ──▶ 顧客PHPサーバー
```

### ポイント

- `data\` は現状の `mock/samba/` に相当。本番では `.env` の `SAMBA_PATH` をこの実フォルダに向けるだけ（コードはそのまま）
- `upload\<型式>\` の構造が、現 mock の `ABC-2021\` 等と同形。Expressはこのフォルダを読んでアップロードする
- `master\`=担当者が手で置く入口、`upload\`=処理の出口、`work\`=処理A/Bの作業場
- Sambaが別NASなら `data\` だけネットワーク先に置く構成もありうるが、運用PCローカルが素直
- ※ `<車種>\<型式>` の切り方や master/work/upload の分け方は、実運用PCの既存構成に合わせて要確認（決まりがあれば config.ts をそれに合わせる）

## 将来性

CLAUDE.mdの「将来の多拠点化」は、この客先デプロイが第一歩。
同一コードを権限で出し分けて各拠点に配置するパターンがそのまま多拠点へ拡張できる。
分離は HTTP 疎結合により低コスト・可逆。
