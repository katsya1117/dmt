# デバッグ方法（層をまたいで追う）

`docs/design/07_アプリケーション方式設計書.md`に書かれている各層（画面→RTK Query→API関数→HTTP→コントローラ→リポジトリ→DB。フックは実ロジックがある時だけ画面とRTK Queryの間に入る）を、実際に1リクエスト分ブレークポイントで追ったり、RTK Queryのキャッシュ状態をRedux DevToolsで視覚的に確認したりする手順。
（作成: 2026-07-14）

---

## 1. VS Codeでフルスタックデバッグを起動する

`.vscode/launch.json`・`.vscode/tasks.json`を用意済み。VS Codeのデバッグパネルで **「Full stack (Server + Client)」** を選んでF5を押すと、以下が同時に立ち上がる。

- **Server (Express/tsx)**: `server`ディレクトリで`npx tsx watch src/index.ts`をデバッグ実行（`tsx watch`のホットリロードにも追従して自動再アタッチする`"restart": true`設定込み）
- **Client (Chrome)**: `preLaunchTask`で`client`の`yarn dev`をバックグラウンド起動した上で、デバッグ用Chromeを`http://localhost:3000`で開く

どちらの構成もVS Code上の`.ts`/`.tsx`ソースに直接ブレークポイントを置けば、ソースマップ経由でそのまま止まる（トランスパイル後のコードを追う必要はない）。

個別に片方だけ使いたい場合は、デバッグパネルの構成選択で「Server (Express/tsx)」または「Client (Chrome)」を単独選択すればよい。

## 2. 層ごとのブレークポイント配置（「アカウント認証を新規追加」の例）

| # | 層 | ファイル:関数 | 何が見えるか |
|---|---|---|---|
| 1 | 画面 | `client/src/pages/AccountAuthTable.tsx` の `handleSubmit` | フォームの生値（`AccountAuthInput`） |
| 2 | RTK Query | `client/src/store/services/accountAuthApi.ts` の `createAccountAuth` エンドポイントの `queryFn` | `records`引数。`createAccountAuth`（api関数）を呼ぶ直前の状態 |
| 3 | API関数 | `client/src/api/accountAuth.ts` の `createAccountAuth` | `http.post`直前のURL・ボディ、`res.data`（axiosの生レスポンス） |
| 4 | HTTPの境界 | ブラウザのNetworkタブ、または後述のXHRブレークポイント | 実際に飛んだJSON・ヘッダ |
| 5 | コントローラ | `server/src/controllers/accountAuthController.ts` の `create` | tsoaが型検証・デシリアライズ**した後**の`body.records`（不正な値はここに来る前に422で弾かれる） |
| 6 | リポジトリ | `server/src/repositories/accountAuth.ts` の `createAccountAuth` | INSERT直前のレコード配列、SQLite側の戻り値 |

戻りも同じ道を逆順で通る。

**重要**：作成が成功すると`invalidatesTags`が効いて、`accountAuthApi.ts`の`accountAuthList`エンドポイント（GET）が自動で再実行される。②〜⑥の一覧取得版（`fetchAccountAuthList`・コントローラの`list`・リポジトリの`listAllAccountAuth`）にもブレークポイントを置いておくと、「更新後に手動リフェッチせず一覧が最新化される」という設計の肝を実際に目撃できる。

### HTTP境界を「自然な壁」として使う

Chrome DevTools（VS Code内蔵Chromeデバッガでも同様）の Sources タブ右側 → **XHR/fetch Breakpoints** に `/api/account-auth` を追加すると、クライアント/サーバーの境界で必ず止まる。個別の関数を探す手間なく「層を跨ぐ瞬間」を機械的に捕まえられるので、初回に流れを掴むにはこちらが楽。

## 3. Redux DevToolsでRTK Query層を読む

RTK Queryのキャッシュ・invalidate・再取得のタイミングは、コードを1行ずつ追うよりもRedux DevTools拡張機能（Chrome/Firefox）で見る方が視覚的。

### State木のどこを見るか

**State** タブで、ストアのルートに`auth`/`jobs`/`localFiles`（普通のslice）と並んで、`accountAuthApi`・`masterApi`というキーがある。これがRTK Queryのキャッシュ全体。

```
accountAuthApi
├─ queries       ← GET相当（useQuery）のキャッシュ本体
├─ mutations     ← POST/PUT相当（useMutation）の実行履歴
├─ provided      ← invalidatesTags/providesTagsの紐付け表（内部管理用）
├─ subscriptions ← 今どのコンポーネントがこのデータを購読中か
└─ config        ← online/focused等の設定値
```

### `queries`の読み方（一番よく見る場所）

キーが`エンドポイント名(引数)`という形式で並ぶ。

```
queries
└─ "accountAuthList(true)"
   ├─ status: "fulfilled"
   ├─ data: [ {id:1, username:"dealer001", ...}, ... ]   ← useAccountAuthListQuery()のdataそのもの
   ├─ endpointName: "accountAuthList"
   ├─ originalArgs: true    ← includeDeletedに渡した値
   └─ fulfilledTimeStamp: 1752...
```

`masterApi.useVehiclesQuery()`なら`masterApi`の`queries["vehicles(undefined)"]`、`useKatashiki('V001')`（`hooks/useMaster.ts`。`skip`条件があるためフック層あり）なら`queries["katashiki(\"V001\")"]`という具合に、**引数ごとに別キーで自動的にキャッシュが分かれている**のがここで目視できる。

### `mutations`の読み方

キーはランダムなrequestIdなので、`originalArgs`（渡した引数）と`endpointName`で見分ける。

```
mutations
└─ "abc123"
   ├─ status: "fulfilled"
   ├─ endpointName: "createAccountAuth"
   ├─ originalArgs: [{ username: "test", password: "..." }]
   └─ data: { inserted: 1 }
```

`mutations`はキャッシュではなく**実行履歴**なので、成功後もしばらく残るが再利用はされない（次にcreateを呼べば新しいrequestIdで別エントリが増える）。

### 実際の操作で追う手順（新規追加の例）

1. DevTools左上の **Action** フィルタ欄に `accountAuthApi` と入力（sliceのnoiseを消す）
2. 「新規追加」ボタンを押す
3. Actionリストに次の順で並ぶ：
   ```
   accountAuthApi/executeMutation/pending      ← create()呼び出し
   accountAuthApi/executeMutation/fulfilled    ← サーバー201が返った
   accountAuthApi/executeQuery/pending         ← invalidatesTagsによる自動refetch開始
   accountAuthApi/executeQuery/fulfilled       ← 一覧が最新化
   ```
4. 各actionをクリックして **Diff** タブを見る（Stateタブより軽い・変化点だけ出る）。`executeMutation/pending`→`fulfilled`で`mutations`に1件増え、続く`executeQuery/pending`→`fulfilled`で`queries["accountAuthList(true)"].data`の配列が1件増えているのが差分として見える
5. `executeMutation/pending`のactionを選び、下の **Action** タブ（Diffの隣）で`meta.arg.endpointName`と`meta.arg.originalArgs`を見ると、「どのエンドポイントに何を渡したか」が生データで確認できる

この4ステップの並びそのものが、「更新後に手動リフェッチせず一覧が自動で最新化される」という設計（`invalidatesTags`）の実体。

### 注意点（このプロジェクト特有）

- `accountAuthList(true)`は現在20,008件あるデモデータなので、Stateタブでツリーを展開すると重い。**Diffタブ中心**で見るか、`data`ノードは畳んだまま`.length`だけ確認するのが安全
- `queries`のキーは`includeDeleted`の値ごとに分かれる（`accountAuthList(true)`と`accountAuthList(false)`は別キャッシュ）。`AccountAuthTable.tsx`は常に`true`で呼んでいるので実質1つしか出ない
- `provided`は普段見なくてよい（`invalidatesTags: ['AccountAuth']`が単純なタグ運用なので、複雑な紐付けを追う場面が今のところ無い）

## 4. 推奨の進め方

1. まずRedux DevToolsで`accountAuthApi`だけにフィルタして一連の流れを1回追い、「どのタイミングで何回通信が起きるか」の全体像を掴む
2. 次にXHRブレークポイントで自然に止め、層をまたぐ瞬間の生データ（リクエスト/レスポンスJSON）を確認
3. 最後に2章の表の各層にブレークポイントを置いて、「Full stack」構成で1リクエストを最初から最後まで1行ずつ追う
