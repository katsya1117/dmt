## サーバー側パース（2026-07-10実装）

### 経緯
クライアント側パース（`exceljs`の`Workbook.xlsx.load`をブラウザで実行）で20000行を読み込むと、ブラウザのメインスレッドが**約10.7秒ブロック**されることを実測で確認した。またこの検証中に、`express.json()`のデフォルトボディ上限(100KB)を超えるリクエストが**エラー内容を一切ログに残さず一律500を返す**バグも発見・修正した（`server/src/index.ts`のエラーハンドラ、`console.error`追加＋実ステータスコード反映）。

### 実装
- パースは`server/src/services/parseAccountAuthExcel.ts`に一本化。①マスタ全件・②差分ファイルとも同じアップロード窓口・同じ経路を通る（画面のボタンは1つのまま）。
- クライアント→サーバーの通信は、JSON化したレコード配列ではなく**ファイルそのものをmultipart(FormData)でアップロード**する方式に変更（`api/accountAuthImport.ts`）。multerの上限は50MBに設定（`server/src/index.ts`の`RegisterRoutes(app, { multer: ... })`。tsoa.jsonの`multerOpts`は非推奨のため使わない）。
- サーバー実測: 20000行のパース＋差分計算で**0.5秒未満**。計測に使ったダミー台帳は`server/test-fixtures/account-auth-import-20000rows.xlsx`にコミット済み（再現手順は同ディレクトリのREADME参照）。

### ストリーミングAPIを断念した経緯（重要）
当初`exceljs`の`stream.xlsx.WorkbookReader`（1行ずつのストリーム読み）で実装したが、**同一バイト列のファイルでも実行するたびに成功/失敗が変わる**という再現性の低いバグに遭遇した：
- curlで直接叩くと安定して成功する
- ブラウザ（axios）経由だと`Cannot read properties of undefined (reading 'sheets')`で高確率で失敗する
- アップロードされたバッファをバイト単位で比較しても内容は完全に同一（`head`/`tail`のhexダンプで確認済み）
- `Readable.from(buffer)`を一時ファイル書き出し＋ファイルパス指定に変えても改善しなかった

原因は`exceljs`のストリーミングリーダー内部で、`workbook.xml`（シート一覧）の解析完了前に`worksheet`エントリの処理が走るケースがある実装上の問題と見られる（`workbook-reader.js`の`_parseWorksheet`が`this.model.sheets`を参照する箇所で発生）。ブラウザ経由のリクエストとcurlとでNode側のストリーム読み取りタイミング（イベントループのスケジューリング）が変わり、再現性に差が出ていたと考えられる。

**対応**: 目的は「ブラウザのメインスレッドを固まらせないこと」であり、サーバー側でNodeのイベントループを数百ms〜1秒止めること自体は、この内部ツール（同時に取り込み操作をするのは実質1人）では許容範囲と判断。安定している非ストリーミングAPI（`Workbook.xlsx.load`、フルパース）を採用し、サーバー側で実行することで当初の目的は達成した（20000行で0.5秒未満、ブラウザで5回連続成功を確認）。

### クライアント側パース(`parseExcel.ts`)を残した理由
`client/src/components/accountAuth/parseExcel.ts`は実運用のフローからは呼ばれなくなったが、**削除せず残している**。理由はStorybook/vitestのMSWモック（`mocks/accountAuthHandlers.ts`）が、実Expressサーバーを起動せずにfetch/axiosを横取りする都合上、ファイルのパースをここに依存しているため。ヘッダー対応や解約日列の判定ルールを変更する際は、こちらとサーバー側`parseAccountAuthExcel.ts`の**両方**を直す必要がある（二重管理のトレードオフを承知の上で採用）。

### 経緯

一覧画面本体（`AccountAuthTable.tsx`）・差分プレビューダイアログ（`ImportDiffDialog.tsx`）の両方をDataGrid化し、他画面（型式選択・約700件等）への横展開テンプレートとした（`docs/画面実装パターン.md`参照）。

### 追記：ページネーションによる検索結果の見落とし対策（2026-07-13）

DataGrid化した直後、**MUI X DataGrid（Community版）はページネーションを完全には無効化できない**（`pagination`プロパティは`true`固定）という制約が判明した。デフォルトのページサイズ（100件）のままだと、「No./ユーザー名で入力した瞬間に該当行が全部見える」という一覧検索の元の設計意図が、**検索結果が100件を超える場合に崩れる**（2ページ目以降をクリックしないと見えない）指摘があり修正した。

対応: `pageSizeOptions`に`{ value: -1, label: 'すべて' }`を追加し、デフォルトのページサイズを`-1`（全件を1ページに表示）にした。これは公式ドキュメントに明記された正式な機能（[MUI公式Pagination解説](https://mui.com/x/react-data-grid/pagination/)で確認済み。node_modulesのソースコードで内部的な挙動を見つけただけの段階では「裏技か正式機能か」を判別できていなかったため、公式ドキュメントで裏取りしてから採用した）。**仮想化はページサイズと独立して効くため、全件を1ページ表示にしても性能は変わらない**（20000行で再計測し、約0.3秒のブロックのまま変化なしを確認）。差分プレビューダイアログ（承認前レビューのため見落とし防止の必要性がより高い）も同様に「すべて表示」固定にした。

## 削除は「行を消す」ではなく「状態を変える」

初期実装では削除済み(delfg=1)行を一覧から除外していたが、これは誤り。**削除は論理削除であり、手動でリストアする手段が無いと詰む**ため、一覧は常に全件（delfg問わず）表示し、「状態」列のチップ（有効／削除済み）で見分ける方針に修正した。削除済み行は編集フォームを開いてdelfgスイッチをOFFに戻せば通常通りリストアできる。

## 解約日列からの削除判定（重要：客先Excelの実態）

客先支給の台帳には、DBの`delfg`に相当する綺麗な0/1フラグ列は無い。実際は**「解約日」列に日付が入っている行が削除対象**で、客先側ではその行を赤く塗りつぶして運用している（2026-07-09に確認）。

- パーサー（`client/src/components/accountAuth/parseExcel.ts`）は「削除フラグ」列 or 「解約日」列のどちらかに値があれば `delfg = true` として扱う。
- 解約日の**値そのものはDBに保存しない**。パース時にbooleanへ変換して捨てる。
- 日付の新旧比較は行わない（実行日以前かどうかのチェックは無し）。理由：業務上、解約日欄に**未来日付が入ることはない**（申込〜DBメンテ依頼まで約1週間のラグがあるだけで、常に過去〜直近の日付）。
- **同じ解約日が何ヶ月台帳に残り続けても、多重処理にはならない**。仕組み上、差分判定は「ファイルから出したdelfg」と「今のDBのdelfg」を突き合わせるだけなので、既に削除済み(DB側delfg=true)の行は、ファイル側delfg=trueと一致し「変更なし」判定になる。「どの日付まで処理済みか」をアプリ側で別途記憶する必要は無い（DBのdelfg自体が処理済みの記憶を兼ねる）。実機で確認済み。

## 既存の重複usernameデータの扱い（2026-07-13追記）

### 業務背景
客先の旧運用では、`username`が重複していても`password`さえ区別されていれば問題ない、という形でデータが作られてきた実績があり、支給される台帳（マスタ全件ファイル）には**現在も現役でエンドユーザーに使われている、username重複データ**が含まれる。これらは変更できない（認証が使われているため）。一方、**今後新規に追加されるアカウントについては、usernameの重複は拒否する**方針とする。

### 対応
- `computeImportDiff`（`server/src/services/accountAuthDiff.ts`）は、**現在のDBで既にusernameが重複している（2件以上存在する）もの**を「レガシー重複」として特定し、ファイルの該当行を**追加/変更/削除/リストアいずれの判定もせず完全にスキップ**する。プレビュー結果の`skippedDuplicateUsernames`にusername一覧が入り、差分プレビューダイアログに警告として表示される
- `validateImportRecords`は、ファイル内でusernameが重複している場合、それが**現在のDBで既に重複しているusernameと一致するなら許容**し（レガシー重複として上記の通りスキップされる）、**一致しない（＝DBにまだ無い、もしくは1件しかないusernameを新規に重複させようとしている）場合のみ拒否する**
- 判定は「DBに既に2件以上あるかどうか」だけを見る（新規追加や変更の可否には一切影響しない、通常のusernameは今まで通り一意性が求められる）

### 検証方法
デモのSQLiteスキーマは`username TEXT NOT NULL UNIQUE`のままのため（本番同様、今後の新規重複は防ぎたいため意図的に維持）、DB内に実際の重複データを作ってブラウザで確認することができない。代わりに`computeImportDiff`・`validateImportRecords`を直接呼び出すスクリプトで4パターン（既存重複の一部更新／新規重複の拒否／既存重複の完全スキップ／通常の新規追加）を検証し、いずれも意図通りの結果になることを確認した（2026-07-13）。

## 背景・業務の流れ

- 得意先からメールが来て、アカウントを **追加 / 削除（論理削除） / 情報変更 / 削除状態からのリストア** する。
- **支給ファイル（Excel台帳）** はその情報が反映された状態で来る建前だが、**間違っていることがある**。
- **間違った更新で客がアカウント認証できなくなる＝大事故。絶対に避ける**（＝安全最優先）。
- 台帳は「秘伝のソース」のように**ずっと同じ1ファイル**で管理（20000行超になることも）。

## 決定事項（この方針で実装する）

### 入力は「3つの窓口」
| # | 窓口 | 入力 | 用途 |
|---|---|---|---|
| ① | **マスタ取り込み** | 全台帳（2万行） | 通常の一括反映 |
| ② | **差分ファイル取り込み** | 手動で切り出した小ファイル | **マスタ破損/怪しい時の安全弁**。必要な行だけ |
| ③ | **手動フォーム**（既存） | 画面入力 | 1件ずつ。**残す** |

### ①と②は「同じ差分レビューエンジン」＝入力サイズが違うだけ
- 差分エンジンは **「ファイルにある行だけ処理」**（無い＝削除にしない）。だから小さい差分ファイルは**自然にその数行だけに効く**（他アカウントは触らない）。
- ②のための新ロジックはほぼ不要。**①の入力を差し替えるだけ**。
- 実装 ＝ 差分レビューエンジン1つ ＋ 入力窓口2つ ＋ 既存フォーム。
- 差分ファイルも**マスタと同じ列**（`delfg`で削除/リストアも表現可）。
- ①マスタ側だけ「大量行だが本当に適用する？」の**確認ガード**を足すと、壊れた全台帳の誤流し込みを防げる。

### 安全上の狙い（②の存在理由）
**マスタファイルが破損していると、それに依存する①だけでは詰む。** メールで来た変更分だけを手で切り出した**小さいクリーンなファイル**を②で流せば、破損マスタに依存せず、影響範囲も小さく（数行）、レビューも一瞬。

### （旧メモ）フォールバック運用
本命（差分レビュー）が難しければ：新規追加=Excel取り込み（軽量ファイル）or手動／更新・削除・復活=手動。
→ 上の②（差分ファイル取り込み）が、この「軽量ファイル」運用も兼ねる。

## 本命：差分レビュー方式（安全最優先）

### 大原則
**自動で適用しない。「差分を見せて → 人が承認 → 適用」。** ファイルは間違いうるので鵜呑みで書き込まない。

### ユーザー操作フロー（3ステップ）
1. メンテナンス > アカウント認証 で **「台帳を取り込む」** を押す（固定パスの台帳をExpressが読む→ファイル選択不要）
2. **差分プレビュー**が出る（追加◯/変更◯/削除◯/リストア◯。★password等の認証系は強調）
3. 内容を確認して **「この内容で適用」** を押す ← ここで初めてDB反映

### 差分の分類（ファイル × 現DBの突合）
| 分類 | 条件 |
|---|---|
| 追加 | ファイルにいて、DBにいない |
| 変更 | 両方にいて、中身が違う |
| 削除 | ファイルで delfg=1 |
| リストア | DB=削除済み、ファイル=有効（delfg 1→0） |
| 変更なし | 中身が同じ（大多数・表示しない） |

### 安全ルール（大事故を防ぐ）
1. **適用前に必ずプレビュー＋承認**（ワンクリック自動適用にしない）。
2. **認証に関わる変更を強調表示**（password / username / delfg）。事故の元。
3. **「ファイルに無い＝削除」にしない**。削除は**明示的に delfg=1 の行だけ**。行が抜け落ちても既存アカウントを壊さない（最大の安全弁）。
4. **元に戻せる**（変更前の値を記録／削除は delfg なので復元可能）。
5. **適用前に検証**（壊れた行・ファイル内重複・必須欠けを弾く）。
6. **upsert 意味論**：既存 username は「エラー」ではなく「更新」。重複エラー地獄を回避。

### 速度・重さ対策（20000行）
- 「遅い」は2つ：①ファイルを読む ②読んだ行を処理する。
- **②処理**：差分なので**変わった数行だけ**検証・書き込み → 激減。人のレビューも差分だけで軽い。
- **①読む**：**サーバー側ストリーミング**（exceljs WorkbookReader で1行ずつ）で実装済み。実測0.44秒。画面を固まらせない目的は達成。進捗表示（SSE等）は今回は見送り、体感で困ってから検討する。
- ただし③として新たな重さが判明：**差分プレビューのテーブル描画**が大量差分（追加20000件等）で仮想化なしのため約13.7秒かかる。パースが軽くなったことで初めて見えたボトルネック。対応は保留中。

## 今のシステム構成への当てはめ

```
[運用PC]
  台帳Excel（固定パス or ブラウザ選択）
    │
  React（取り込み画面）
   ①preview要求 ─POST /api/account-auth/import/preview─▶ Express(tsoaコントローラ新規)
                                                          ②exceljsストリーム読み
                                                          ③repositories/accountAuthで現DB読む
                                                          ④突合→差分分類（書き込まない）
   ◀── 差分JSON ───────────────────────────────────────┘
   ⑤差分レビュー表（★認証系強調）
   ⑥apply ─POST /api/account-auth/import/apply─▶ Express
                                                  承認行だけ repository経由で upsert/delfg
   ◀── 結果 ──┘
```

- **パースをクライアント→Expressへ移す**（現 `client/components/accountAuth/parseExcel.ts` はブラウザ全読み。台帳はサーバーストリームが本命。少数の臨時取り込み用にクライアントパースを残す選択肢はあり）。
- **repositories が差し替え点**：突合も適用も repository 経由。今SQLite、本番は客先DB/PHP。**画面・契約は無変更**。
- **tsoa で preview/apply の2エンドポイントを型から生成** → クライアントに差分型が自動で降りる。
- **MSWモック**で preview/apply を用意 → Storybook/テストは Express 無しで差分UI確認。
- **論理削除(delfg)** がそのまま「削除/リストア」と「復元可能性」に効く。

### PHP は書き足すのか（本番）
- 取り込みの**差分・レビューはExpress**。PHPに新業務ロジックは増えない。
- 実データ操作は **list / create / update / delfg更新** ＝**単票CRUDと同じ**。旧FLEXアプリが既にやっていた操作なので、業務ロジックは概ね AMFPHP に存在。
- 必要になるのは「**既存操作をRESTで露出**（AMF→REST ポート）」＝新機能ではなく既存の露出。取り込み専用の特別なPHPは不要。
- **今（デモ）は repository=SQLite なので PHP ゼロで preview/apply とも動く**。

## 実装の入口（明日以降）

安全で分かりやすい第一歩＝**preview（差分を計算して画面に出すだけ・書き込みなし）** から。
1. サーバー：取り込み用コントローラ（tsoa）＋差分計算（exceljsストリーム＋repository読み）→ preview 返却
2. クライアント：取り込み画面（差分レビュー表・認証系強調）
3. その後 apply（承認→upsert/delfg）＋検証＋履歴（元に戻せる）
4. 並行して手動追加（既存）は維持。フォールバック運用（軽量ファイル新規追加＋他は手動）も想定に含める。
