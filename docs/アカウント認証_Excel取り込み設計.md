## サーバー側パース（2026-07-10実装）

### 経緯
クライアント側パース（`exceljs`の`Workbook.xlsx.load`をブラウザで実行）で20000行を読み込むと、ブラウザのメインスレッドが**約10.7秒ブロック**されることを実測で確認した。またこの検証中に、`express.json()`のデフォルトボディ上限(100KB)を超えるリクエストが**エラー内容を一切ログに残さず一律500を返す**バグも発見・修正した（`server/src/index.ts`のエラーハンドラ、`console.error`追加＋実ステータスコード反映）。

### 実装
- パースは`server/src/services/parseAccountAuthExcel.ts`に一本化。①マスタ全件・②差分ファイルとも同じアップロード窓口・同じ経路を通る（画面のボタンは1つのまま）。
- クライアント→サーバーの通信は、JSON化したレコード配列ではなく**ファイルそのものをmultipart(FormData)でアップロード**する方式に変更（`api/accountAuthImport.ts`）。multerの上限は50MBに設定（`server/src/index.ts`の`RegisterRoutes(app, { multer: ... })`。tsoa.jsonの`multerOpts`は非推奨のため使わない）。
- サーバー実測: 20000行のパース＋差分計算で**0.5秒未満**。計測に使ったダミー台帳は`server/test-fixtures/account-auth-import-20000rows.xlsx`にコミット済み（再現手順は同ディレクトリのREADME参照）。

### ストリーミングAPIを断念した経緯（重要）
当初`exceljs`の`stream.xlsx.WorkbookReader`（1行ずつのストリーム読み）で実装したが、**同一バイト列のファイルでも実行するたびに成功/失敗が変わる**という再現性の低いバグに遭遇した：
- curlで直接叩くと安定して成功する
- ブラウザ（axios）経由だと`Cannot read properties of undefined (reading 'sheets')`で高確率で失敗する
- アップロードされたバッファをバイト単位で比較しても内容は完全に同一（`head`/`tail`のhexダンプで確認済み）
- `Readable.from(buffer)`を一時ファイル書き出し＋ファイルパス指定に変えても改善しなかった

原因は`exceljs`のストリーミングリーダー内部で、`workbook.xml`（シート一覧）の解析完了前に`worksheet`エントリの処理が走るケースがある実装上の問題と見られる（`workbook-reader.js`の`_parseWorksheet`が`this.model.sheets`を参照する箇所で発生）。ブラウザ経由のリクエストとcurlとでNode側のストリーム読み取りタイミング（イベントループのスケジューリング）が変わり、再現性に差が出ていたと考えられる。

**対応**: 目的は「ブラウザのメインスレッドを固まらせないこと」であり、サーバー側でNodeのイベントループを数百ms〜1秒止めること自体は、この内部ツール（同時に取り込み操作をするのは実質1人）では許容範囲と判断。安定している非ストリーミングAPI（`Workbook.xlsx.load`、フルパース）を採用し、サーバー側で実行することで当初の目的は達成した（20000行で0.5秒未満、ブラウザで5回連続成功を確認）。

### クライアント側パース(`parseExcel.ts`)を残した理由
`client/src/components/accountAuth/parseExcel.ts`は実運用のフローからは呼ばれなくなったが、**削除せず残している**。理由はStorybook/vitestのMSWモック（`mocks/accountAuthHandlers.ts`）が、実Expressサーバーを起動せずにfetch/axiosを横取りする都合上、ファイルのパースをここに依存しているため。ヘッダー対応や解約日列の判定ルールを変更する際は、こちらとサーバー側`parseAccountAuthExcel.ts`の**両方**を直す必要がある（二重管理のトレードオフを承知の上で採用）。

### 追記：ページネーションによる検索結果の見落とし対策（2026-07-13→2026-07-14方針転換）

DataGrid化した直後、**MUI X DataGrid（Community版）はページネーションを完全には無効化できない**（`pagination`プロパティは`true`固定）という制約が判明した。デフォルトのページサイズ（100件）のままだと、「No./ユーザー名で入力した瞬間に該当行が全部見える」という一覧検索の元の設計意図が、**検索結果が100件を超える場合に崩れる**（2ページ目以降をクリックしないと見えない）指摘があり修正した。

**2026-07-13時点の対応**：`pageSizeOptions`に`{ value: -1, label: 'すべて' }`を追加し、デフォルトのページサイズを`-1`（全件を1ページに表示）にした。仮想化はページサイズと独立して効くため、全件を1ページ表示にしても性能は変わらないことを確認済み（20000行で約0.3秒のブロックのまま変化なし）。

**2026-07-14方針転換**：この対応では、ページネーションUI自体は残っていたため、**ユーザーがRows per pageを25/50/100に選び直せてしまい、同じ「絞り込んだのに何件か見えない」罠を再現できる状態**が残っていた。onChangeで即時絞り込み・常に全件表示が前提の画面ではページという概念自体が不要と判断し、`AccountAuthTable`一覧画面は`hideFooter`でページネーションUIごと非表示にした（総件数は`Typography`で別途表示）。**差分プレビューダイアログ（`ImportDiffDialog.tsx`）は検索機能が無く元々ページ送りの罠が起きないため、`pageSizeOptions`による全件表示固定のままで問題ない**（`hideFooter`化はしていない）。

**落とし穴と修正（同日）**：`hideFooter`導入時、`initialState`の`pageSize: -1`指定を誤って削除してしまい、**ページネーションUIは消えたがページネーション自体は生きたまま**になり、デフォルトの100件しか表示されない不具合が発生した（`hideFooter`はUI表示のみを制御し、ページング動作自体は無効化しない）。`initialState={{ pagination: { paginationModel: { pageSize: -1 } } }}`を`hideFooter`と併用する形に修正し、実機でDataGridの`aria-rowcount`（20012＝ヘッダー1+データ20011）を確認して全件表示になっていることを検証した。

## 削除は「行を消す」ではなく「状態を変える」

初期実装では削除済み(delfg=1)行を一覧から除外していたが、これは誤り。**削除は論理削除であり、手動でリストアする手段が無いと詰む**ため、一覧は常に全件（delfg問わず）表示し、「状態」列のチップ（有効／削除済み）で見分ける方針に修正した。削除済み行は編集フォームを開いてdelfgスイッチをOFFに戻せば通常通りリストアできる。

## 解約日列からの削除判定（重要：客先Excelの実態）

客先支給の台帳には、DBの`delfg`に相当する綺麗な0/1フラグ列は無い。実際は**「解約日」列に日付が入っている行が削除対象**で、客先側ではその行を赤く塗りつぶして運用している（2026-07-09に確認）。

- パーサー（`client/src/components/accountAuth/parseExcel.ts`）は「削除フラグ」列 or 「解約日」列のどちらかに値があれば `delfg = true` として扱う。
- 解約日の**値そのものはDBに保存しない**。パース時にbooleanへ変換して捨てる。
- 日付の新旧比較は行わない（実行日以前かどうかのチェックは無し）。理由：業務上、解約日欄に**未来日付が入ることはない**（申込〜DBメンテ依頼まで約1週間のラグがあるだけで、常に過去〜直近の日付）。
- **同じ解約日が何ヶ月台帳に残り続けても、多重処理にはならない**。仕組み上、差分判定は「ファイルから出したdelfg」と「今のDBのdelfg」を突き合わせるだけなので、既に削除済み(DB側delfg=true)の行は、ファイル側delfg=trueと一致し「変更なし」判定になる。「どの日付まで処理済みか」をアプリ側で別途記憶する必要は無い（DBのdelfg自体が処理済みの記憶を兼ねる）。実機で確認済み。

## 既存の重複usernameデータの扱い（2026-07-13追記）

### 業務背景
客先の旧運用では、`username`が重複していても`password`さえ区別されていれば問題ない、という形でデータが作られてきた実績があり、支給される台帳（マスタ全件ファイル）には**現在も現役でエンドユーザーに使われている、username重複データ**が含まれる。これらのusernameは変更できない（認証が使われているため）が、**それ以外のカラム（備考・住所等）は通常のレコードと同様に更新されうる**。一方、**今後新規に追加されるアカウントについては、usernameの重複は拒否する**方針とする。

### 対応：No.のハードコードによる明示的な除外リスト（2026-07-13、方式変更／2026-07-14、照合方式を修正）
当初は「現在のDBでusernameが2件以上あれば自動的にレガシー扱い」という動的判定で実装したが、**この方式だと将来何らかの理由で意図しない新しい重複が発生した場合もサイレントに見逃してしまう**という弱点が判明し、方式を変更した。

- `server/src/services/accountAuthDiff.ts`の`LEGACY_DUPLICATE_NUMBERS`に、**既知の問題レコードのNo.を明示的に列挙**する。実体は`server/src/data/accountAuthExcelKnownNumbers.json`の`legacyDuplicateNumbers`配列（2026-07-16、envではなく専用JSONデータファイルに分離。理由は下記「設定の持ち方」参照）。2026-07-13時点、実際のNo.は客先確認待ちで空の配列
- `computeImportDiff`は、ファイルの各行の`number`（No.）がこのリストに含まれる場合、**usernameではなくNo.でDB行を照合**する。**usernameの重複チェックが対象外になるだけで、対応するDB行が見つかれば他カラムの追加/変更/削除/リストア判定は通常のレコードと同じように行われる**（2026-07-14修正。以前はusernameで一意特定できないという理由でこのNo.のレコードを丸ごとスキップしており、他カラムの変更も一切反映されないバグがあった。No.は一意という前提のもと照合キーをusernameからNo.に切り替えて解決した）
- No.に対応するDB行が万一見つからない場合（データ不整合）のみ、変更を保留し`skippedDuplicateUsernames`に記録され、差分プレビューダイアログに警告として表示される。**この記録もusernameでdedupしない**（`{username, number}`を行ごとにそのまま持つ）。同じusernameで複数のNo.が未マッチになるケースがあり得るため、usernameだけで集約すると「どのNo.が問題か」が分からなくなる（2026-07-14修正。ユーザー指摘により発覚。No.が本来の一意なキーである以上、表示用の集計もNo.ベースにする必要があった）
- `validateImportRecords`も同じリストを参照し、既知のNo.以外でusernameが重複している行のみ拒否する。エラーメッセージも**行番号を含める**（`X行目：usernameがY行目と重複しています`）。以前はusernameしか出しておらず、同じusernameが3件以上あった場合にファイルの何行目同士が重複しているか特定できなかった（2026-07-16修正。上記`skippedDuplicateUsernames`と同種の問題としてユーザー指摘）
- この方式だと、**リストに載っていない未知の重複が万一発生した場合は、通常の差分（追加/変更）として人間のレビューに乗る**ため、サイレントに見逃されることがない（動的判定より安全側に倒れている）
- 適用処理（`server/src/repositories/accountAuth.ts`の`applyAccountAuthImport`）も、以前は`WHERE username = ?`で更新対象を特定していたため、username重複行に対しては「どちらの行が更新されるか不定」という別のバグを抱えていた。2026-07-14、`computeImportDiff`が特定したDB行の`id`を使う方式に変更し、この曖昧さも解消した（レガシー重複に限らず全レコードが対象）
- クライアント側のMSWモック（`client/src/mocks/accountAuthHandlers.ts`）にも同名の仕組みを実装し、Storybook/テストでも同じ挙動を再現している（デモなのでリストは空のまま）

### 設定の持ち方：envではなく専用JSONデータファイル（2026-07-16）
`LEGACY_DUPLICATE_NUMBERS`・`KESSABAN_NUMBERS`（後述）は当初TypeScriptファイル内に直書きの配列としていたが、**環境変数（env）ではなく専用のJSONデータファイルに分離**した。

- **envにしなかった理由**：これらは実行環境（開発/本番）ごとに変わる設定ではなく、**客先の実データに紐づく業務データ**。envは接続先URLやパスなど環境差分を表現するためのもので、数十件になりうる数値リストや、コードレビューで差分を追いたい業務データを持たせるのには向かない
- **実体**：`server/src/data/accountAuthExcelKnownNumbers.json`（`legacyDuplicateNumbers`・`kessabanNumbers`の2配列）。`resolveJsonModule`でTypeScriptから直接importする（`tsc`ビルド時に`dist/data/`へ自動コピーされることを確認済み）
- **クライアント側の鏡**：`client/src/data/accountAuthExcelKnownNumbers.json`に同じ内容を複製（MSWモック用。サーバーとクライアントは別パッケージのため直接共有はできず、既存のロジック複製と同じ考え方で両方揃える運用にした）。実際のNo.が判明した際は**両方のJSONファイルを更新すること**
- コード（`.ts`ファイル）を一切触らずにNo.を追加・削除できるため、客先確認後の反映がしやすい

### 検証方法
デモのSQLiteスキーマは`username TEXT NOT NULL UNIQUE`のままのため（本番同様、今後の新規重複は防ぎたいため意図的に維持）、DB内に実際の重複データを作ってブラウザで確認することができない。代わりに`computeImportDiff`・`validateImportRecords`を直接呼び出すスクリプトで、`LEGACY_DUPLICATE_NUMBERS`に一時的にテスト用のNo.を入れて検証した。2026-07-14の修正では、同一usernameでNo.違いの2件（一方だけ備考を変更）を用意し、①バリデーションでusername重複エラーにならない、②No.が一致する側だけが`changed`として検知されfield単位で正しく差分が出る、③No.が一致するがフィールドが同じ側は`unchangedCount`に数えられる（＝No.ベースの照合が正しく機能している証拠）、④通常レコードも従来通り動く、の4点を確認した（検証後、`LEGACY_DUPLICATE_NUMBERS`は空配列に戻してある）。

## 「←欠番」等の結合セル注記行の除外（2026-07-14実装→同日撤回→2026-07-16実データ確認の上で復活）

### 経緯
客先の台帳では、No.を欠番にした行を「←欠番」のような注記付きの**結合セル**で表現している箇所がある、という想定のもと、一時**自作のテストファイル**（A〜C列を結合しusername列まで「←欠番」を巻き込ませた想定で作成）で検証した。exceljsは結合セルの値を結合範囲内の全セルに複製して返すため、そのテストファイルでは「←欠番」という文字列がusername・password・commentすべてに入り込み、空行として除外されずに新規登録扱いになる現象を確認し、`KESSABAN_NUMBERS`（No.ハードコード方式）による対策を一度実装した。

### 一度撤回した理由
これは**実際の客先ファイルではなく自作テストファイルの想定（結合範囲がusername列まで及ぶ）に基づく対策**であり、客先ファイルの実際の結合パターンは未確認のままだった。「実際の『欠番』行はusername列が空欄で、埋まっているのはNo.列だけではないか」という仮説のもと、YAGNI（実データで確認できていない問題への対策を先回りして作り込まない）の観点から、`KESSABAN_NUMBERS`とその関連ロジックを一度撤去し、既存の空行フィルタのみに一本化した（2026-07-14）。

### 実データで確認の上、復活（2026-07-16）
その後、**実際の客先ファイルでusername列まで結合が巻き込むことが確認された**ため、`KESSABAN_NUMBERS`（`server/src/services/parseAccountAuthExcel.ts`・`client/src/components/accountAuth/parseExcel.ts`の両方）を復活させた。動作確認は、実データと同じ構造（A〜C列結合＋「←欠番」、No.列は独立）のテストファイルを作り、`KESSABAN_NUMBERS`に一時的にテスト用No.を入れて「対象行が除外され、前後の正常行は影響を受けない」ことを確認した。実体は`server/src/data/accountAuthExcelKnownNumbers.json`の`kessabanNumbers`配列（前述「設定の持ち方」参照）。実際のNo.は客先確認待ちで空の配列（TODO）のまま。

## 他テーブルへの展開の雛形

この画面（差分プレビューのテーブル表示・区分チップ・認証系フィールド強調・適用ボタン・絞り込み検索・削除は状態チップで表現）は、**今後CSV取り込み等を含め他テーブルに同種の機能を展開する際の雛形**として設計している。個別最適化せず、普遍的・汎用的・直感的なUI/UXを意識すること。

## 背景・業務の流れ

- 得意先からメールが来て、アカウントを **追加 / 削除（論理削除） / 情報変更 / 削除状態からのリストア** する。
- **支給ファイル（Excel台帳）** はその情報が反映された状態で来る建前だが、**間違っていることがある**。
- **間違った更新で客がアカウント認証できなくなる＝大事故。絶対に避ける**（＝安全最優先）。
- 台帳は「秘伝のソース」のように**ずっと同じ1ファイル**で管理（20000行超になることも）。

## 決定事項（この方針で実装する）

### 入力は「3つの窓口」
| # | 窓口 | 入力 | 用途 |
|---|---|---|---|
| ① | **マスタ取り込み** | 全台帳（2万行） | 通常の一括反映 |
| ② | **差分ファイル取り込み** | 手動で切り出した小ファイル | **マスタ破損/怪しい時の安全弁**。必要な行だけ |
| ③ | **手動フォーム**（既存） | 画面入力 | 1件ずつ。**残す** |

### ①と②は「同じ差分レビューエンジン」＝入力サイズが違うだけ
- 差分エンジンは **「ファイルにある行だけ処理」**（無い＝削除にしない）。だから小さい差分ファイルは**自然にその数行だけに効く**（他アカウントは触らない）。
- ②のための新ロジックはほぼ不要。**①の入力を差し替えるだけ**。
- 実装 ＝ 差分レビューエンジン1つ ＋ 入力窓口2つ ＋ 既存フォーム。
- 差分ファイルも**マスタと同じ列**（`delfg`で削除/リストアも表現可）。
- ①マスタ側だけ「大量行だが本当に適用する？」の**確認ガード**を足すと、壊れた全台帳の誤流し込みを防げる。

### 安全上の狙い（②の存在理由）
**マスタファイルが破損していると、それに依存する①だけでは詰む。** メールで来た変更分だけを手で切り出した**小さいクリーンなファイル**を②で流せば、破損マスタに依存せず、影響範囲も小さく（数行）、レビューも一瞬。

### （旧メモ）フォールバック運用
本命（差分レビュー）が難しければ：新規追加=Excel取り込み（軽量ファイル）or手動／更新・削除・復活=手動。
→ 上の②（差分ファイル取り込み）が、この「軽量ファイル」運用も兼ねる。

## 本命：差分レビュー方式（安全最優先）

### 大原則
**自動で適用しない。「差分を見せて → 人が承認 → 適用」。** ファイルは間違いうるので鵜呑みで書き込まない。

### ユーザー操作フロー（3ステップ）
1. メンテナンス > アカウント認証 で **「台帳を取り込む」** を押す（固定パスの台帳をExpressが読む→ファイル選択不要）
2. **差分プレビュー**が出る（追加◯/変更◯/削除◯/リストア◯。★password等の認証系は強調）
3. 内容を確認して **「この内容で適用」** を押す ← ここで初めてDB反映

### 差分の分類（ファイル × 現DBの突合）
| 分類 | 条件 |
|---|---|
| 追加 | ファイルにいて、DBにいない |
| 変更 | 両方にいて、中身が違う |
| 削除 | ファイルで delfg=1 |
| リストア | DB=削除済み、ファイル=有効（delfg 1→0） |
| 変更なし | 中身が同じ（大多数・表示しない） |

### 安全ルール（大事故を防ぐ）
1. **適用前に必ずプレビュー＋承認**（ワンクリック自動適用にしない）。
2. **認証に関わる変更を強調表示**（password / username / delfg）。事故の元。
3. **「ファイルに無い＝削除」にしない**。削除は**明示的に delfg=1 の行だけ**。行が抜け落ちても既存アカウントを壊さない（最大の安全弁）。
4. **元に戻せる**（変更前の値を記録／削除は delfg なので復元可能）。
5. **適用前に検証**（壊れた行・ファイル内重複・必須欠けを弾く）。
6. **upsert 意味論**：既存 username は「エラー」ではなく「更新」。重複エラー地獄を回避。

### 速度・重さ対策（20000行）
- 「遅い」は2つ：①ファイルを読む ②読んだ行を処理する。
- **②処理**：差分なので**変わった数行だけ**検証・書き込み → 激減。人のレビューも差分だけで軽い。
- **①読む**：**サーバー側ストリーミング**（exceljs WorkbookReader で1行ずつ）で実装済み。実測0.44秒。画面を固まらせない目的は達成。進捗表示（SSE等）は今回は見送り、体感で困ってから検討する。
- ただし③として新たな重さが判明：**差分プレビューのテーブル描画**が大量差分（追加20000件等）で仮想化なしのため約13.7秒かかる。パースが軽くなったことで初めて見えたボトルネック。対応は保留中。

## 今のシステム構成への当てはめ

```
[運用PC]
  台帳Excel（固定パス or ブラウザ選択）
    │
  React（取り込み画面）
   ①preview要求 ─POST /api/account-auth/import/preview─▶ Express(tsoaコントローラ新規)
                                                          ②exceljsストリーム読み
                                                          ③repositories/accountAuthで現DB読む
                                                          ④突合→差分分類（書き込まない）
   ◀── 差分JSON ───────────────────────────────────────┘
   ⑤差分レビュー表（★認証系強調）
   ⑥apply ─POST /api/account-auth/import/apply─▶ Express
                                                  承認行だけ repository経由で upsert/delfg
   ◀── 結果 ──┘
```

- **パースをクライアント→Expressへ移す**（現 `client/components/accountAuth/parseExcel.ts` はブラウザ全読み。台帳はサーバーストリームが本命。少数の臨時取り込み用にクライアントパースを残す選択肢はあり）。
- **repositories が差し替え点**：突合も適用も repository 経由。今SQLite、本番は客先PHPのAPI呼び出し（DBを直接操作するのはPHPのみ）。**画面・契約は無変更**。
- **tsoa で preview/apply の2エンドポイントを型から生成** → クライアントに差分型が自動で降りる。
- **MSWモック**で preview/apply を用意 → Storybook/テストは Express 無しで差分UI確認。
- **論理削除(delfg)** がそのまま「削除/リストア」と「復元可能性」に効く。

### PHP は書き足すのか（本番）
- 取り込みの**差分・レビューはExpress**。PHPに新業務ロジックは増えない。
- 実データ操作は **list / create / update / delfg更新** ＝**単票CRUDと同じ**。旧FLEXアプリが既にやっていた操作なので、業務ロジックは概ね AMFPHP に存在。
- 必要になるのは「**既存操作をRESTで露出**（AMF→REST ポート）」＝新機能ではなく既存の露出。取り込み専用の特別なPHPは不要。
- **今（デモ）は repository=SQLite なので PHP ゼロで preview/apply とも動く**。

## 実装の入口（明日以降）

安全で分かりやすい第一歩＝**preview（差分を計算して画面に出すだけ・書き込みなし）** から。
1. サーバー：取り込み用コントローラ（tsoa）＋差分計算（exceljsストリーム＋repository読み）→ preview 返却

4. 並行して手動追加（既存）は維持。フォールバック運用（軽量ファイル新規追加＋他は手動）も想定に含める。
