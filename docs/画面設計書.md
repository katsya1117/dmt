# 画面設計書 — d社マニュアルメンテナンスツール

> 目的: 工数見積もりのため、成果物（=作る画面・部品）の解像度を上げる。
> 本書は **クライアント（React）側の画面** を定義する。API・データ設計は別資料。

## 前提（確定事項）

| 項目 | 決定 |
|------|------|
| 利用環境 | 社内LAN内・**認証なし**（ログイン画面は作らない） |
| 同時利用 | 想定少数（運用PC中心）。排他制御は当面なし |
| アップロードUI | **一本化**（型式を選ぶと全種別を一括表示・一括アップロード） |
| ナビゲーション | サイドバー型 |
| 処理A/B | 今回スコープ外。画面は枠だけ用意し「準備中」表示 |

## 画面一覧

| # | 画面 | 役割 | 今回 | 備考 |
|---|------|------|:----:|------|
| S1 | アップロード | 型式を複数選択 → 対象ファイル確認 → 一括アップロード → 結果/再実行 | ◯ | **本ツールの主画面** |
| S2 | 履歴 | アップロードジョブの一覧・詳細ログ | ◯ | 一括操作が増えるほど重要 |
| S3 | 設定 | 接続設定（Sambaパス/PHP URL）・ファイル種別設定 | ◯ | `config.ts` のGUI化 |
| S4 | 処理（将来） | 処理A/Bの起動・進捗 | △ | 枠＋「準備中」のみ。501想定 |

> 認証なしのため **ログイン画面（S0）は不要**。

## 画面遷移

```
        ┌──────────────────────────────────────┐
        │  共通レイアウト（サイドバー + ヘッダー）  │
        └──────────────────────────────────────┘
   サイドバー
     ├─ [アップロード] ─→ S1 ──(実行)──→ 結果表示（同一画面内）
     │                          └─(履歴を見る)─→ S2
     ├─ [履歴]        ─→ S2 ──(行クリック)──→ ジョブ詳細（モーダル/サブ画面）
     ├─ [設定]        ─→ S3（タブ: 接続設定 / ファイル種別）
     └─ [処理(将来)]   ─→ S4（準備中）
```

画面遷移はサイドバー切替が基本。S1の結果表示・S2のジョブ詳細は **同一画面内の領域 or モーダル** で完結（別ルートを増やさない）。

## S1: アップロード画面（主画面）

### 目的
「**どの型式の・どのファイルを・どこへ**上げるか」を可視化し、複数型式を一括アップロード。失敗時は失敗分だけ再実行。

### ワイヤーフレーム
```
┌─────────────────────────────┬───────────────────────────────────┐
│ ① 型式選択                    │ ② 対象ファイル確認                  │
│ [🔍 検索______]  車種▾ 年式▾   │ アップロード先: //server/.../{型式}  │
│ ☑ ABC-2021                   │ ┌─ ABC-2021 ──────────────┐        │
│ ☑ ABC-2022                   │ │ ⚠ 型式別DB なし           │        │
│ ☐ XYZ-2020                   │ │ XML  manual_001.xml  12KB │        │
│ ...（約700・仮想スクロール）   │ │ PDF  manual_001.pdf  …    │        │
│ [全選択] [解除]  選択: 2件     │ └──────────────────────────┘        │
│                              │ ┌─ ABC-2022 ──────────────┐        │
│                              │ │ DB  katashiki.db  ...     │        │
│                              │ │ SWF manual_001.swf …      │        │
│                              │ └──────────────────────────┘        │
├──────────────────────────────┴───────────────────────────────────┤
│ ③ 実行バー:  読み取り元 //samba  →  送信先 PHP {url}   [アップロード]│
├────────────────────────────────────────────────────────────────────┤
│ ④ 進捗 / 結果                                                        │
│ ABC-2021 ████████ 成功(4)   ABC-2022 ███░░ 失敗(1) [この型式を再実行]│
│  └ manual_001.swf : 保存失敗 (502)                                  │
└────────────────────────────────────────────────────────────────────┘
```

### 構成部品と見積り観点

| ID | 部品 | 機能 | 状態 | 依存API |
|----|------|------|------|---------|
| S1-1 | 型式選択リスト | 一覧表示・複数選択・検索・車種/年式フィルタ・全選択 | 読込中/空/エラー。**約700件→仮想スクロール必須** | `GET /api/katashiki` |
| S1-2 | 対象ファイル確認パネル | 選択型式ごとにファイルをグルーピング表示（種別・名前・サイズ・アップ先） | 型式追加で逐次読込。**型式別DB未含有を警告表示** | `GET /api/katashiki/:id/files` |
| S1-3 | 実行バー | 読取元(Samba)→送信先(PHP)を明示。一括アップロード起動 | 選択0件で無効。実行中は再押下不可 | — |
| S1-4 | 進捗・結果 | 型式単位の進捗バー・成否サマリ・ファイル単位エラー | 待機/転送中/成功/部分成功/失敗 | `POST /api/upload/:id`（型式ごと） |
| S1-5 | 再実行 | 失敗した型式/ファイルのみ再アップロード | 失敗ジョブが存在する時のみ表示 | `POST /api/upload/:id` |

> **見積り注意点**
> - 一括アップロードは型式ごとに `POST /api/upload/:id` を **逐次/並列で複数回** 呼ぶ設計（現状APIは型式単位）。同時実行数の制御ロジックが必要。
> - 「型式別DB未含有の警告」（CLAUDE.md/メモの最重要ドメイン知識）はフロントで判定 or APIに含める要検討 → **API側設計に影響**。
> - 約700型式の一覧は仮想スクロール/フィルタ前提（DOM全描画は不可）。

### アップロードジョブの状態遷移
```
待機 ──実行──→ 転送中 ──┬─ 全成功 ─→ 成功
                       ├─ 一部失敗 ─→ 部分成功 ──再実行──→ 転送中
                       └─ 全失敗 ─→ 失敗 ────────再実行──→ 転送中
```

## S2: 履歴画面

### 目的
過去のアップロード結果を確認。失敗の追跡・監査。一括操作が増えるほど重要（確定要件）。

### ワイヤーフレーム
```
┌───────────────────────────────────────────────────────────┐
│ フィルタ: [日付▾] [型式🔍] [結果▾(成功/部分/失敗)]            │
├───────┬──────────┬────────┬───────┬───────┬──────────────┤
│ 日時   │ 型式      │ ファイル │ 結果   │ 所要   │              │
├───────┼──────────┼────────┼───────┼───────┼──────────────┤
│ 06/17 │ ABC-2021 │ 4       │ 成功   │ 3.2s  │ [詳細]        │
│ 06/17 │ ABC-2022 │ 5       │ 部分   │ 5.1s  │ [詳細]        │
└───────┴──────────┴────────┴───────┴───────┴──────────────┘
   ↓ [詳細]
┌─ ジョブ詳細（モーダル）──────────────┐
│ ABC-2022 / 06/17 14:20               │
│ ✓ manual_001.xml   ✓ katashiki.db    │
│ ✗ manual_001.swf  502 保存失敗        │
│ [この失敗分を再実行]                  │
└──────────────────────────────────────┘
```

### 構成部品

| ID | 部品 | 機能 | 依存API（**新規**） |
|----|------|------|---------------------|
| S2-1 | 履歴一覧 | ジョブ一覧・フィルタ・ソート | `GET /api/history`（要新設） |
| S2-2 | ジョブ詳細 | ファイル単位の成否ログ | `GET /api/history/:jobId`（要新設） |
| S2-3 | 履歴からの再実行 | 失敗分のみ再アップロード | `POST /api/upload/:id` |

> **見積り注意点**: 履歴は **サーバ側に永続化が必要**（現状APIに履歴保存なし）。保存先（SQLite/JSONファイル等）の決定が前提。→ API・データ設計資料へ申し送り。

## S3: 設定画面

### 目的
`server/src/config.ts` の内容をGUIで編集。SWF廃止等の変更に備える（設計方針）。

### タブ構成
```
[ 接続設定 ] [ ファイル種別設定 ]

― 接続設定 ―
  Sambaパス      [ ../mock/samba                    ] [接続テスト]
  PHPサーバーURL  [ http://localhost:8080           ] [接続テスト]

― ファイル種別設定 ―（resourceTypes の編集）
  拡張子 | ラベル          | アップ先ディレクトリ |
  xml    | XML（分割済み）  | xml                  | [削除]
  pdf    | PDF             | pdf                  | [削除]
  swf    | SWF             | swf                  | [削除]
  ...                                       [＋種別を追加]
                                                  [保存]
```

### 構成部品

| ID | 部品 | 機能 | 依存API（**新規**） |
|----|------|------|---------------------|
| S3-1 | 接続設定フォーム | Sambaパス/PHP URLの表示・編集・接続テスト | `GET/PUT /api/config`, `POST /api/config/test`（要新設） |
| S3-2 | ファイル種別エディタ | resourceTypesの一覧・追加・編集・削除 | `GET/PUT /api/config`（要新設） |

> **見積り注意点**: 現状 `config.ts` はソース直書き＋環境変数。GUI編集にするなら **設定の永続化（ファイル書き出し）と再読込** の仕組みが必要。「読み取り専用表示のみ」にすれば工数は大幅減 → **要確認スコープ**。
